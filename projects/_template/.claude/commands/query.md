# /query — поиск + синтез

> Формат страниц — в `wiki/_schema.md`, нужен только если будешь сохранять synthesis.

Аргумент: вопрос на естественном языке.

## Шаг 0 — локальный индекс

1. Если `.reefiki/index.sqlite` отсутствует или явно устарел — выполнить из корня репо:
   `python scripts/reefiki.py --project projects/<name> index`.
2. Выполнить поиск:
   `python scripts/reefiki.py --project projects/<name> search "<вопрос>" --limit 7`.
3. Если скрипт недоступен — fallback через `wiki/index.md`:
   - Прочитать `wiki/index.md` целиком.
   - Найти записи, у которых `tags` или `useful_when` содержат слова из запроса (case-insensitive).
   - Для каждого совпадения прочитать заголовок и `## Суть` из файла (не грузить целиком).
   - Отсортировать по релевантности (совпадение в `useful_when` > совпадение в `tags` > совпадение в `title`).

Не объяснять этот шаг пользователю.

## Шаг 1 — progressive disclosure

1. Сначала показать короткий ответ по top 3–7 страницам.
2. В ответе всегда указывать provenance: `страница → источник/log`, например `(→ wiki/concepts/x.md; source: raw/y.md)`.
3. Полные страницы читать только для кандидатов, которые реально нужны ответу.
4. Если пользователь просит детали — раскрыть следующий слой: история/таймлайн через `python scripts/reefiki.py --project projects/<name> timeline --limit 10`.
5. Если нужна полная страница — открыть конкретный файл, не грузить всю wiki.

## Шаг 2 — синтез ответа

- Чётко ответить на вопрос.
- Цитировать конкретные страницы: `(→ wiki/concepts/<slug>.md)` или `[[<slug>]]`.
- Для каждой ключевой опоры указывать provenance: `page`, `sources`, при необходимости `wiki/log.md`.
- Если страницы противоречат — отметить явно, сослаться на блок `## Conflicting claims`.
- Если ответа в wiki нет — сказать прямо. Не выдумывать.
- Если для ответа нужна информация из `raw/` — прочитать соответствующий файл и процитировать как `(source: raw/<filename>)`.

## Шаг 3 — после ответа

1. **Обновить телеметрию** (батчится, не коммитится сразу):
   - `use_count`: +1 в каждой использованной странице.
   - `last_used`: обновить только если `null` ИЛИ старше сегодняшней даты -1 день (т.е. не чаще раза в 24ч). Это режет шум от повторных запросов.
2. Если ответ — синтез из 2+ страниц: предложить:
   `Сохранить этот синтез как wiki/synthesis/<slug>.md? (Y/N)`
   При Y — создать страницу с frontmatter `type: synthesis`, обновить `index.md`. Формат — в `wiki/_schema.md`.
3. **Gap detection.** Если ответа в wiki нет — после «не нашлось» добавить: «Если найдёшь ответ — скажи мне, я сохраню как страницу». Это сигнал пользователю, что пробел можно закрыть.
4. Append в лог: `## [YYYY-MM-DD] /query | "<запрос>" → N pages used`. Если gap — `→ gap`.
5. **Логировать релевантность** (метрики агента): в той же log-строке добавить `| relevant: Y` если ответ был полезным (нашлось что-то по теме), или `| relevant: N` если нашлось, но не то / пришлось ответить «не знаю». Также добавить `| provenance: Y`, если в ответе были page/source ссылки.
   Пример: `## [2026-05-01] /query | "как настроить sync" → 3 pages used | relevant: Y`.
   **Зачем:** если `relevant: N` ≥ 30% запросов за месяц → триггер для Phase 1 (qmd-поиск).
6. **Коммитить только если был synthesis-save** (`git commit -m "<project>: query saved synthesis/<slug>"`). Иначе оставить всё в working tree — изменения телеметрии будут включены в ближайший `/process`/`/lint`/`/harvest`. Это плановый батчинг: один коммит вместо десятков.
