# /harvest — захват знаний из сессии

> Формат новых страниц — `wiki/_schema.md` (читай только если будешь создавать записи).

Самый ценный источник — живой опыт работы, а не чужие репо. Эта команда фиксирует его.

## Контекст

Проанализировать содержимое **текущей сессии диалога** (или явного описания от пользователя). Найти три типа сигналов.

## Быстрый режим

Если пользователь явно вызвал `/harvest`, «запомни», «зафиксируй», «сохрани важное» или подтвердил ранее предложенный harvest, не жди отдельного подтверждения списка кандидатов. Выполни bounded harvest сразу.

Bounded harvest:
- сохранить только durable-важное, обычно 1-3 страницы за проход;
- не запускать `memory pack`, `golden`, deep audit, full-project scan или handoff;
- читать только `wiki/index.md`, релевантные страницы и явно названные пользователем/сессией файлы;
- если материала больше чем на 3 страницы, сохранить первые 1-3 самые полезные и назвать лучший следующий harvest-слой.

## Подготовка

Прочитать `wiki/index.md` — что уже есть.

Если сессия была длинной или была dev-задача:

1. Посмотреть `plans/*.md`, если есть активный план.
2. Посмотреть последние записи: `python scripts/reefiki.py --project projects/<name> timeline --limit 10`. Если скрипт недоступен — прочитать последние 10 записей из `wiki/log.md` вручную (записи начинаются с `## [`).
3. Если активного плана нет, но задача длинная/многошаговая — создать план:
   `python scripts/reefiki.py --project projects/<name> plan create "<короткое имя задачи>"`.
4. В wiki переносить только проверенные решения, процедуры и выводы. Сам plan-файл — рабочая память, не долговременная wiki.

## Три типа сигналов

**РЕШЕНИЯ** — «мы выбрали X вместо Y потому что Z».
→ кандидат для `wiki/decisions/` (тип `decision`, ADR-формат).

**ОТКРЫТИЯ** — «оказывается, X работает так-то», «выяснили, что Y».
→ кандидат для `wiki/synthesis/` (тип `synthesis`).

**ПАТТЕРНЫ** — повторяющийся подход, приём, конфигурация.
→ кандидат для `wiki/concepts/` (тип `concept`).

**НАВЫКИ** — проверенная пошаговая процедура: «как проверять», «как настраивать», «как чинить».
→ кандидат для `wiki/skills/` (тип `skill`).

**Авто-детекция skill:**
- Агент выполнил одну и ту же последовательность действий 2+ раз в сессии → предложить skill.
- Пользователь сказал «запомни как делать» / «сохрани процедуру» → сразу skill.
- Успешно отлажен баг / настроен инструмент / найден воркараунд → предложить skill.

## Сверка

Для каждого кандидата: есть ли уже что-то в INDEX? Если да — это апдейт существующей страницы (расширить, добавить дату), не новая.

## Acceptance check для synthesis

Перед предложением `wiki/synthesis/` задай внутренний фильтр: «чем эта страница будет полезна через 3 месяца?»

- Если ответ — только «это снимок текущей сессии» или «история отладки», не предлагай synthesis.
- Если материал сырой, но ценный и объёмный — предложи `deep-research/<topic>/`.
- Если это повтор, временный лог, одноразовый статус или доказательство без процедуры — отложи в `seen/` с причиной.
- Если есть применимый вывод, решение или процедура — переформулируй как `concept`, `decision` или `skill`, а synthesis используй только для действительно связанной выжимки.

## Вывод кандидатов

```text
Из этой сессии я бы сохранил:

[1] РЕШЕНИЕ: использовать X вместо Y для Z → wiki/decisions/use-x-over-y.md
    Дельта: первый ADR по этой теме

[2] ОТКРЫТИЕ: паттерн retry с jitter работает только при N>3 → wiki/synthesis/retry-jitter.md
    Дельта: уточняет существующий [[retry-logic]]

[3] ПАТТЕРН: claude-cli + obsidian web clipper → ускоряет capture x5 → wiki/concepts/web-clipper-pipeline.md
    Дельта: в INDEX нет

[4] НАВЫК: проверка .env перед деплоем (rights 600, key presence, /v1/models) → wiki/skills/check-env-before-deploy.md
    Дельта: воспроизводимая процедура, не факт

[5] — больше ничего достойного сохранения
```

Если harvest предложен агентом проактивно, а пользователь ещё не подтвердил — жди ответа: `всё / 1 и 3 / только 1 / ничего`.

Если harvest явно requested пользователем — не жди ответа, сразу выполняй bounded harvest.

> Если сессия была короткой и ничего не найдено — скажи коротко: «Ничего достойного сохранения не нашлось».

## После подтверждения

1. Создать выбранные страницы (frontmatter обязателен, `useful_when` заполнить).
2. Расставить `[[links]]` на связанные существующие страницы.
3. Append в лог: `## [YYYY-MM-DD] /harvest | N entries (decisions/X, concepts/Y, synthesis/Z, skills/W)`.
4. Обновить `wiki/index.md`.
5. Пересобрать локальный индекс: `python scripts/reefiki.py --project projects/<name> index`.
6. Если пользователь попросил commit/push или это уже принятый поток текущей задачи, stage только затронутые файлы harvest и commit. Не делай широкий `git add -A`, если в repo есть чужие dirty changes.
7. Для cross-project harvest из подключённого кодового проекта используй корневую команду:
   `python scripts/reefiki.py harvest-commit --target-project <name> --path <wiki-file> ... --message "<msg>"`.
   Она коммитит через изолированный git index, поэтому чужие staged/dirty файлы из других проектов не попадут в harvest commit. Dirty вне target project — это excluded context, не blocker; blocker только path вне `projects/<name>/wiki/**` или уже staged target-файл.

## Логирование trigger-событий (метрики агента)

Каждый раз, когда агент **предлагает** `/harvest` по авто-триггеру (не по явной просьбе пользователя), логировать итог:

- Пользователь принял (сохранил ≥1 запись): в лог добавить строку `- trigger: harvest-proposed | accepted (N entries)`.
- Пользователь отказался (`ничего` / закрыл без выбора): в лог добавить строку `- trigger: harvest-proposed | declined`.

Строка добавляется **внутри** записи текущего `/harvest` (или отдельной mini-записью если harvest не состоялся).

**Зачем:** если `declined` ≥ 3 подряд для одного типа триггера — предложить пользователю ослабить этот триггер в `AGENTS.md`.
