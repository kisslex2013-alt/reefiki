# /process — extraction pass

> Если будешь создавать страницы — прочитай `wiki/_schema.md` (один раз за сессию).

## Подготовка

1. Прочитать `wiki/index.md` целиком (для дельта-анализа).
2. Прочитать `_domain.md` если ещё не читал в этой сессии.
3. Получить список файлов в `inbox/`.

## source_id (стабильный идентификатор)

Каждый источник — URL, локальный файл или путь к репо — имеет стабильный `source_id` для дедупликации против `wiki/` и `seen/`.

Для URL — нормализация перед сравнением:

- lowercase host
- убрать `?query` и `#fragment`
- убрать trailing `/`
- убрать `www.`

Для локального файла — имя без пути и расширения, kebab-case.

Для репо — `<owner>-<repo>` для GitHub/GitLab/Bitbucket, `<repo>` для локального пути.

`source_id` сравнивается с `id` существующих страниц и с `source` в `seen/*.md`. Совпадение → SEEN с `reason: duplicate`, `duplicate_of: <slug>`.

## Safety-фильтр (defence in depth)

Перед разбором выполнить из корня репо:
`python scripts/reefiki.py --project projects/<name> privacy`.
Если проверка падает — сначала показать опасные файлы и не обрабатывать их.

Перед обработкой каждого файла применить тот же фильтр, что в `/save` (см. `save.md` секция «Safety-фильтр»). Если файл попал в `inbox/` мимо `/save` (например, скопирован руками) — пропусти его и скажи пользователю: «`<имя>` похоже на `<тип>`, не обрабатываю. Удали из копилки или переименуй».

Не путать с обычным «вне домена» — там запрос пользователя, тут — техническое запрещённое содержимое.

## Для каждого файла

### Распознай тип источника

1. **Репозиторий** — URL вида `https?://github.com/<owner>/<repo>` (или `gitlab.com`, `bitbucket.org`), либо локальный путь, который выглядит как git-репо (содержит `.git/`, или README + один из `package.json` / `requirements.txt` / `Cargo.toml` / `go.mod` / `pom.xml` / `*.gemspec`). → **Репо-режим** (см. ниже).
2. **URL-стаб** — любой другой URL без сохранённого контента.
3. **Контент** — файл с текстом (md/txt/pdf-извлечение/clip и т.п.).

### URL-стаб (не репо)

- Веб-доступ есть → получить контент → сохранить в `raw/<source-id>.md` (markdown с заголовком `# <title>` и блоком метаданных URL/date/fetched-at в начале) → обработать как Контент.
- Веб-доступа нет → дай пользователю **конкретную инструкцию**: «Открой `<url>` через Obsidian Web Clipper или скопируй содержимое в файл и сохрани в `inbox/<slug>.md`. Затем запусти `/process` снова». Файл оставить в `inbox/`.

### Репо-режим

Цель: создать одну `entity`-страницу про репо. Не разбирать как обычный текст.

**Получить содержимое:**

- Локальный путь — читай напрямую.
- GitHub URL — попробуй `gh repo view <owner/repo> --json name,description,languages,topics,readme`. Если `gh` нет — `git clone --depth 1 <url> <temp_dir>` во временную папку, после чтения удалить. Нет сети — скажи пользователю по-человечески: «Не могу достать репо из интернета. Скачай локально и дай путь».

**Читай только метафайлы** (в порядке приоритета, пропускай несуществующие):

1. `README.md` / `README.rst` / `README.txt`
2. `package.json` / `requirements.txt` / `Cargo.toml` / `go.mod` / `pom.xml` / `*.gemspec`
3. `docker-compose.yml` / `Dockerfile`
4. Имена папок верхнего уровня (только имена, не содержимое)
5. `.env.example` или очевидный главный конфиг

Не читай исходный код (`.js`, `.py`, `.ts` и т.д.) — только метафайлы.

**Внутренние 7 вопросов** (вместо стандартных 4 для контента):

1. **ЗАЧЕМ** — какую проблему решает (1–2 предложения, без жаргона).
2. **СТЕК** — языки, фреймворки, ключевые зависимости.
3. **КАК ДУМАЕТ** — архитектурный паттерн, главная идея структуры.
4. **ПРИНЦИПЫ** — что нельзя нарушать, зафиксированные design-решения.
5. **ИНТЕГРАЦИИ** — внешние API, сервисы, связанные репо.
6. **СТАТУС** — `active` / `maintenance` / `archived` / `prototype`.
7. **НЕОЧЕВИДНОЕ** — что нужно знать, чтобы не сломать.

**Анти-дубликат:** прочитай `wiki/index.md`. Если репо уже есть — спроси: «Страница уже есть, обновить?»

**Страница** — `wiki/entities/<repo-slug>.md`. Формат frontmatter — в `wiki/_schema.md`. Тело страницы:

```markdown
## Зачем
<1–2 предложения>

## Стек
<язык, фреймворки, ключевые зависимости>

## Как думает
<архитектурный подход, главный паттерн — 1 абзац>

## Принципы
<design decisions, что нельзя нарушать>

## Интеграции
<с чем взаимодействует>

## Статус
<active | maintenance | archived | prototype> — <последний коммит если знаешь>

## Неочевидное
<что нужно знать, чтобы не сломать>

## Related
<[[связанные страницы]] из wiki, если нашёл>
```

**Правила репо-режима:**

- НЕ копировать README целиком — только дистиллят.
- НЕ описывать файлы пофайлово.
- Временный клон удалить после чтения, не коммитить.
- Если репо явно вне `_domain.md` — спроси, как для обычного контента.

После создания страницы — общий поток: сводка пользователю, обновить `index.md`, лог-запись `## [YYYY-MM-DD] /process | repo <repo-name> → entities/<slug>`, коммит.

### Контент

**Большие источники.** Если файл/URL >50KB или >10000 слов — не читай целиком сразу:

1. Прочитай только структуру: оглавление, заголовки разделов, первые/последние абзацы каждого раздела (outline-проход).
2. По outline реши, какие разделы нужны для ответа на 4 вопроса ниже. Остальные пропусти.
3. Прочитай выбранные разделы и дистиллируй. В `raw/` сохраняется **полный** файл, в wiki-страницу идёт только дистиллят релевантных разделов.

Иначе — прочитать полностью. Применить **внутренние 4 вопроса** (не показывай пользователю — они твой reasoning):

1. **Дельта.** Что уникального относительно `wiki/index.md`?
2. **Сценарий.** При каком конкретном сценарии станет нужным? (формирует `useful_when`)
3. **Тип.** `source` / `entity` / `concept` / `synthesis` / `decision` / `skill`?
4. **Решение.** SAVE или SEEN, одна фраза причина.

#### Триггер на `seen/` без полного анализа

Если файл явно дубль (тот же URL/title/основные тезисы что у существующей страницы) — сразу SEEN, не трать время на полный анализ.

#### Если `useful_when` не выводится

См. `wiki/_schema.md` секцию «Правило валидности».

#### Запрос вне домена

Если контент явно не вписывается в `_domain.md` — спроси пользователя: «Это вне темы проекта `<NAME>`. Сохранить здесь или предложить создать новый проект?» Не сохраняй молча.

## Acceptance check (последний фильтр перед SAVE)

Перед каждым SAVE — три внутренних вопроса. На любой нет конкретного ответа → SEEN с `reason: low-value` или `unclear-use`. Не «может пригодиться», не общие формулировки.

1. **Что взять.** Какой конкретный фрагмент/тезис уйдёт в `## Суть` и `## Дельта`?
2. **Когда открыть.** Какой конкретный сценарий применения попадёт в `useful_when` (то же, что вопрос «Сценарий» выше)?
3. **Чем отличается от того, что уже есть.** Что нового относительно текущих страниц в `wiki/index.md`? Если только пересказ существующей — `seen/` с `reason: duplicate` или `low-value`.

Этот фильтр работает поверх 4 вопросов и `useful_when`-валидности из `_schema.md`. Цель — не дать «архивации» победить «дистилляцию».

## Сводка перед записью

Покажи пользователю **по-человечески** (без терминов `frontmatter`, `useful_when`, `slug`):

```text
Из копилки готовлю:

[1] Карпати про LLM-вики → сохраняю как страницу о методе персональной памяти LLM
    Применение: когда буду проектировать память агента или сравнивать с RAG

[2] Чей-то перепост Карпати → отложу как повтор: ничего нового по сравнению с (1)

[3] URL без контента → нужна твоя помощь, см. инструкцию выше
```

Спроси: «Применить? (всё / только 1 / отмена)»

## После подтверждения

Для каждого SAVE:

1. Скопировать файл `inbox/` → `raw/<source-id>.md` (kebab-case). Для URL-стабов файл в `raw/` уже создан на шаге URL-стаб (fetched content) — пропустить.
2. Создать `wiki/<type>s/<id>.md` (формат — в `wiki/_schema.md`). `id` = `source-id`. Для `skill` — `wiki/skills/<id>.md`.
3. Если уместно — создать/обновить связанные страницы. Перед каждой — анти-дубликат-проверка по `wiki/`: ищи существующие страницы с тем же `id`/`title`/тегом-концептом.
4. Расставить `[[Related]]` в секции `## Related` тела страницы (если есть смежные).
5. Обновить `wiki/index.md`.

Для каждого SEEN: создать `seen/<source-slug>.md` с `quarantine_until` = сегодня + 30 дней.

Удалить обработанные файлы из `inbox/`.

## Конфликты

Если новый источник противоречит существующей странице — **не перезаписывай**. Добавь секцию `## Conflicting claims` (формат — в `_schema.md`). В лог: `## [YYYY-MM-DD] conflict | <page> vs <source>`.

## Финал

1. Append в `wiki/log.md`: `## [YYYY-MM-DD] /process | N saved, M to seen, K skipped`.
2. Обновить `wiki/index.md` если нужно.
3. Пересобрать локальный индекс: `python scripts/reefiki.py --project projects/<name> index`.
4. Добавить в stage только явные пути, затронутые этим process run: созданные/обновлённые wiki-страницы, `wiki/index.md`, `wiki/log.md`, перемещённые/удалённые inbox-файлы, созданные seen-файлы, созданные raw-файлы и пересобранные `.reefiki` artifacts если они были; затем `git commit -m "<project>: process inbox (N saved, M to seen)"`. Будущий машинный guard должен использовать process operation profile, а не broad staging.
5. Итог пользователю по-человечески:

   ```text
   Готово. Сохранил: N. Отложил как повторы: M. Нужна твоя помощь: K (см. выше).
   Копилка пуста.
   ```

6. **Авто-проверка**: посчитай в `log.md` сколько `/process` после последнего `/lint`. Если ≥10 → предложи `/lint`.
