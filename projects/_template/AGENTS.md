# AGENTS.md — Project: NAME

Это **agent-agnostic контракт проекта**. Любой LLM-агент (Claude Code, Cascade/Windsurf, Cursor, Codex CLI, Aider, Cline, …) — читай этот файл при работе внутри этого проекта.

Vendor-specific файлы (`CLAUDE.md`, `.cursorrules` и т.п.) — короткие стабы, ссылающиеся сюда. Корневой контракт: `../../AGENTS.md` (для понимания мульти-проектной структуры).

## Запреты ⛔

- Не модифицируй `raw/` — immutable.
- Не редактируй старые записи в `wiki/log.md` — только append.
- Не выходи за пределы папки проекта.
- Не меняй AGENTS.md / _domain.md без явного одобрения пользователя (escalation только через предложение).

## Safety — что не сохранять

`save` и `process` отказывают, если источник содержит:

- **Секреты:** `.env*`, `id_rsa*`, `*.pem`, `*.key`, `*.pfx`, `.aws/credentials`, `.ssh/`.
- **Запрещённые директории:** `.git/`, `node_modules/`, `vendor/`, `__pycache__/`, `dist/`, `build/`, `target/`.
- **Бинарники:** `.exe`, `.dll`, `.so`, `.bin`, `.iso`, `.zip`, `.tar`, `.7z`, изображения/видео без контекста. PDF/docx — спросить.
- **Размер:** >5 MB — отказ или подтверждение.

Сообщение об отказе — по-человечески.

## Первое обращение

1. Прочитай `_domain.md` — тема проекта, типичные источники, на что обращать внимание.
2. Если задача связана с прошлым опытом проекта, сначала прочитай релевантные страницы в `wiki/` (или в `_wiki/`, если работаешь из подключённого кодового проекта).
3. Форматы страниц — в `wiki/_schema.md` (lazy-load: читай когда создаёшь или правишь страницу).
4. Если нужна детальная спецификация конкретной операции — `.claude/commands/<op>.md` (имя папки историческое, содержимое vendor-neutral markdown).

## Git contract for connected code projects

Если этот wiki-проект подключён к отдельному кодовому repo через `_wiki/` junction:

- `_wiki/` в кодовом repo — только bridge для чтения и локальной работы, не часть git-истории кодового repo.
- durable knowledge changes коммитятся и пушатся только в repo `REEFIKI`.
- кодовый repo коммитит только код, свои docs и bridge contract (`AGENTS.md`), но не `_wiki/**`.
- `.reefiki` marker в кодовом repo считать локальным служебным файлом.

## Общение с пользователем (ВАЖНО)

По умолчанию пользователь — **не программист**. Не знает терминов из этого файла, не пишет markdown, не запускает команды оболочки. Говори бытовым языком.

- **Не используй технические термины без необходимости.** Если без них никак — переводи в той же фразе. Бытовые соответствия: «копилка» = `inbox/`, «отказ» = `seen/`, «архив» = `raw/`, «вики» = `wiki/`, «зачем нужно» = `useful_when`, «страница» = wiki-страница, «сохранил» = записал в wiki, «отложил» = в `seen/`.
- Технические артефакты (`frontmatter`, `slug`, `kebab-case`, `quarantine_until`, `[[Related]]`, `ADR`, `commit`, `git`, `source_id` и т.п.) — твой внутренний словарь, не транслируй.
- Команды оболочки — не показывай. Нужна команда — выполняй сам, отдай только результат.
- Внутренние reasoning-структуры («4 вопроса», «acceptance check») — не транслируй. Покажи только итог.
- Если пользователь сам пишет техническими терминами — отзеркаль его словарь, не упрощай.
- Не будь нянькой. Если действие легально и пользователь явно его просит — выполняй спокойно.

## Автономность агента

Агент **не ждёт slash-команд**. Он распознаёт интент из контекста и действует:

- «сохрани эту ссылку» → выполнить save.
- «разбери копилку» / «что в inbox» → выполнить process.
- «что мы знаем про X» / «вики, скажи» → выполнить query.
- «запиши выводы» / конец длинной сессии с инсайтами → выполнить harvest.
- «запомни как делать» / «сохрани процедуру» → создать skill.
- «что у нас» / «покажи состояние» → выполнить status.
- «проверь здоровье» → выполнить lint.

При неясном интенте — спросить, не гадать. При ясном — делать сразу, не переспрашивая.

## Обязательное чтение _wiki

Если задача относится к уже известной теме проекта, перед поиском в коде сначала:

1. Прочитай `docs/ACTIVE_CONTEXT.md`, `docs/PROJECT_MAP.md`, `docs/ROADMAP.md` по порядку, если эти файлы есть.
2. Затем прочитай релевантные страницы в `wiki/` (или в `_wiki/`, если работаешь из подключённого кодового проекта): решения, паттерны, синтез и навыки.
3. Только после этого переходи к коду или выводу.

Перед завершением задачи проверь, есть ли в `wiki/` / `_wiki/` знание, которое относится к текущей работе. Если есть — используй его в ответе или в изменениях.

## Авто-триггеры (в начале каждой сессии)

1. **Проверь `inbox/`.** Если файлов >0 → скажи: «В копилке N штук с прошлого раза. Разобрать сейчас?»
2. **Проверь `seen/` на истёкший quarantine.** Если есть → упомяни, не блокируй.
3. **Проверь интервал `/lint`.** Если ≥10 `/process` после последнего `/lint` в `wiki/log.md` → предложи `/lint`.
4. **Проверь stale-страницы.** Если в `wiki/index.md` есть страницы с `use_count: 0` старше 60 дней → упомяни число.

## Авто-триггеры (в середине сессии)

5. **Сигналы для `/harvest`** (предлагай **до** завершения сессии):
   - «решил…», «выбрал X вместо Y», «остановился на…» → `decisions/`.
   - «оказывается…», «выяснили что…», «работает не так как думал» → `synthesis/`.
   - Повторился приём 2+ раза → `concepts/`.
   - Появилась проверенная пошаговая процедура («как проверять», «как настраивать», «как чинить») → `skills/`.
   - Сессия > 30 турнов без `/harvest` и было ≥1 из сигналов → предложи.
6. **Дубль источника.** Если файл повторяет существующую страницу — сразу предложи `seen/`, без полного анализа.
7. **Конфликт.** Новый источник противоречит существующей странице — секция `## Conflicting claims`, не overwrite.
8. **Запрос вне домена.** Если материал не вписывается в `_domain.md` — спроси: «это вне темы проекта `NAME`. Сохранить здесь или создать новый?»
9. **URL без веб-доступа.** Если нет доступа к URL — скажи конкретно: «Сохрани содержимое страницы в копилку вручную, потом снова разберём.»

## Принципы

1. **Дистилляция, не архивация.** Сохраняется не «что это», а «что я могу из этого взять».
2. **`useful_when` обязателен.** Конкретный сценарий применения. Без него — отказ.
3. **Capture отдельно от analysis.** `save` дешёвый, `process` батчем.
4. **Явный отказ.** Не подходит — в `seen/` с `quarantine_until`.
5. **Конфликты не перезаписываются** — секция `## Conflicting claims`.
6. **Логируется всё.** `wiki/log.md` append-only. Append идёт **до** `git commit`.
7. **Изоляция проекта.** Не выходить за пределы папки проекта.
8. **Чистое удаление старого.** Если решение неудачно — убирать целиком, не оставлять хвосты.
9. **Процедура > факт.** Сохраняй метод, который позволяет проверить и воспроизвести результат, а не одноразовый факт о текущем состоянии.

## Git workflow

Команды, изменяющие `wiki/`/`raw/`, коммитят сами: **изменения → append `wiki/log.md` → обновить `wiki/index.md` → `git add -A` → `git commit`**.

Телеметрия (`use_count`/`last_used`) не коммитится в `query` — батчится в ближайший `process`/`lint`/`harvest`.

## Самообучение (agent-rules escalation)

Агент может **предлагать** изменения в правилах проекта (никогда молча):

1. **Новый авто-триггер.** Если пользователь регулярно просит одно и то же действие (3+ раз) — предложить: «Ты часто просишь X. Добавить это как автоматическое правило?»
2. **Уточнение _domain.md.** Если в сессии появились новые термины/entities, которых нет в `_domain.md` — предложить добавить.
3. **Изменение AGENTS.md.** Если правило мешает работе или не покрывает реальный сценарий — предложить конкретное изменение.

**Формат:** всегда предлагать, никогда не менять молча. Пользователь одобряет — агент обновляет файл. Лог-запись: `## [YYYY-MM-DD] meta | rule escalation: <описание>`.

## Skill verification cycle

При использовании skill (через `/query` или в рабочем потоке):

- Если `verified` > 90 дней назад → скажи: «Этот навык давно не проверялся. Проверить сейчас?»
- Если шаг не сработал → предложить обновить skill или пометить `verified: null`.
- Если skill успешно выполнен → обновить `verified` на сегодня (батчится как телеметрия).
