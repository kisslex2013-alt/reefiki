## 1. Конфликты поведения: реальные и мнимые

### Где конфликт действительно возникает

Есть три слоя инструкций которые агент читает одновременно:

```
Приоритет (сверху вниз как агент их обрабатывает):
┌─────────────────────────────────────────────────┐
│ 1. System prompt самой IDE                      │
│    (Anthropic/OpenAI/Google настраивают агента) │
├─────────────────────────────────────────────────┤
│ 2. AGENTS.md / CLAUDE.md — твои правила         │
│    (ты пишешь сам)                              │
├─────────────────────────────────────────────────┤
│ 3. _user/profile.md — предпочтения пользователя │
│    (wizard + агент дополняет)                   │
└─────────────────────────────────────────────────┘
```

Конфликт возникает когда одно и то же правило описано на двух уровнях с разным значением. Конкретные сценарии:

**Сценарий A — прямое противоречие по одному параметру**

```yaml
# AGENTS.md (ты написал):
"Всегда показывай diff перед изменением wiki-страницы"

# _user/profile.md (wizard записал):
autonomy: high   # = делай и сообщай, без подтверждений
```

Агент получает два несовместимых правила. Одни агенты берут последнее прочитанное, другие — первое, третьи пытаются совместить и делают что-то непредсказуемое.

**Сценарий B — разный язык при одном смысле**

```yaml
# AGENTS.md:
"Объясняй подробно, с примерами кода"

# _user/profile.md:
explanation_style: concise
```

Технически это конфликт, но менее опасный — агент скорее всего выберет один стиль, просто непонятно какой.

**Сценарий C — IDE-специфичные ограничения vs _user/**

Claude Code имеет встроенные правила про безопасность, ограничения на запись файлов, поведение при ошибках. Они живут в system prompt самого IDE и **не видны тебе**. `_user/profile.md` с `autonomy: high` не может их переопределить — и не должен пытаться.

---

### Как это решить: иерархия с явными доменами

Главный принцип: **каждый слой отвечает за свою область, не пересекается с другими**.

```
AGENTS.md          = ЧТО делать (операции, workflow, запреты)
_user/profile.md   = КАК делать (стиль, тон, автономность)
IDE system prompt  = МОЖНО ли делать вообще (безопасность, ограничения)
```

Это разделение нужно зафиксировать явно в самом `AGENTS.md` — тогда агент понимает что `_user/` не конкурирует с операционными правилами, а дополняет их:

```markdown
## Приоритет инструкций

Правила читаются в следующем порядке. При конфликте побеждает
правило из более высокого слоя.

1. Ограничения IDE (system prompt) — абсолютный приоритет, не обсуждается
2. AGENTS.md — операционные правила REEFIKI (что делать, как работать с wiki)
3. _user/profile.md — стиль и предпочтения (как общаться, насколько автономно)

Слой 3 управляет ТОЛЬКО:
  - тоном и стилем объяснений
  - языком общения
  - уровнем автономности В РАМКАХ того что разрешено слоем 2
  - частотой уточняющих вопросов

Слой 3 НЕ может переопределять:
  - операционные запреты из AGENTS.md
  - требования показывать diff перед записью
  - governance правила (useful_when обязателен, quarantine и т.д.)
  - любые правила безопасности
```

---

### Практическое разделение: что куда писать

Вот конкретная граница — что остаётся в AGENTS.md и что идёт в `_user/`:

| Правило | AGENTS.md | `_user/profile.md` |
|---|---|---|
| Всегда проверять frontmatter перед записью | ✅ | ❌ |
| Показывать diff перед изменением wiki | ✅ | ❌ |
| Quarantine 30 дней — не менять | ✅ | ❌ |
| Не публиковать private проекты | ✅ | ❌ |
| Язык ответов (ru/en/zh) | ❌ | ✅ |
| Краткость vs подробность объяснений | ❌ | ✅ |
| Спрашивать подтверждение или нет | Граница | Уточнение |
| Уровень технических объяснений | ❌ | ✅ |

Строка "спрашивать подтверждение" — граница. AGENTS.md определяет **когда обязательно** спрашивать (перед необратимыми операциями). `_user/profile.md` определяет **в остальных случаях** — спрашивать или нет:

```markdown
## В AGENTS.md (фиксированное):
Обязательно подтверждение перед:
  - publish-task --apply
  - удалением из wiki/
  - разрешением конфликтов

## В _user/profile.md (настраиваемое):
autonomy: high
# = в остальных случаях делай без вопросов
```

---

### Решение для CLAUDE.md конкретно

CLAUDE.md сейчас — тонкий stub: одна строка указывающая на AGENTS.md. Добавить вторую строку:

```markdown
See AGENTS.md for operational rules.
See _user/profile.md for communication style preferences.
When rules conflict, AGENTS.md takes precedence over _user/.
```

Это явная декларация приоритета — Claude читает оба файла и знает что делать при конфликте.

## 2. Расход токенов: цифры и оптимизация

### Сколько добавляет `_user/`

Реальный размер файлов при типичном наполнении:

| Файл | Токенов при старте | Частота чтения |
|---|---|---|
| `_user/profile.md` | ~150–200 | Каждая сессия |
| `_user/context.md` | ~100–150 | Каждая сессия |
| `_user/patterns.md` | ~200–400 (растёт) | Только high confidence |
| `_user/log.md` | 0 | Не читается при старте |
| **Итого добавка** | **~450–750 токенов** | |

Для сравнения: AGENTS.md весит ~3000–5000 токенов. `_user/` добавляет **10–15% к текущему bootstrap**. Это приемлемо.

Но есть нюанс: `_user/patterns.md` растёт со временем. Через полгода активного использования он может вырасти до 1500–2000 токенов. Это управляемо, но требует стратегии.

### Три уровня оптимизации

**Уровень 1 — Immediate: компактный формат (бесплатно)**

Большинство полей в profile.md можно сжать без потери смысла. Агент читает структуру, не эссе:

```yaml
# Многословно (150 токенов):
---
type: user-profile
schema_version: "1.0"
created: 2026-06-04
updated: 2026-06-04
wizard_mode: quick
---

## Коммуникация
language: ru
explanation_style: concise
autonomy: high

## Технический контекст
expertise_self_reported: advanced
primary_stack: [Python, git, SQLite]
current_ides: [claude-code, codex, cursor]
```

```yaml
# Компактно (60 токенов):
---
lang: ru | style: concise | autonomy: high
expertise: advanced | ides: [claude-code, codex, cursor]
stack: [Python, git, SQLite]
---
```

Разница ~60% токенов при той же смысловой нагрузке для агента.

**Уровень 2 — Patterns pruning: автоматическая очистка**

`patterns.md` без управления раздувается. Добавить правило в AGENTS.md:

```markdown
## _user/patterns.md — управление размером

При добавлении нового наблюдения:
- Если patterns.md > 800 токенов: найти запись с confidence: low
  старше 60 дней → пометить [архив] и убрать из активного чтения
- Максимум активных записей: 15
- [архив] записи не читаются при bootstrap, хранятся для истории
```

Это cap на размер — файл растёт, но активная часть остаётся в пределах.

**Уровень 3 — Compiled profile: один файл вместо четырёх**

По аналогии с `drift-cache.json` из предыдущих разговоров — скрипт компилирует `_user/` в один компактный файл который агент читает при старте:

```python
# scripts/reefiki.py — новая команда
def compile_user_context(reefiki_path: Path) -> Path:
    """
    Компилирует _user/ в один компактный файл для быстрого чтения агентом.
    Запускается автоматически при изменении любого файла в _user/.
    """
    profile = parse_frontmatter((_user / "profile.md").read_text())[0]
    context = parse_frontmatter((_user / "context.md").read_text())[0]
    patterns_text = (_user / "patterns.md").read_text()

    # Только high-confidence паттерны
    high_conf = [
        line for line in patterns_text.splitlines()
        if "confidence: high" in line or
           ("confidence: medium" in line and "[архив]" not in line)
    ]

    compiled = {
        "lang": profile.get("lang", "ru"),
        "style": profile.get("style", "concise"),
        "autonomy": profile.get("autonomy", "high"),
        "expertise": profile.get("expertise", "intermediate"),
        "ides": profile.get("current_ides", []),
        "focus": context.get("current_goal", ""),
        "active": context.get("active_projects", []),
        "patterns": high_conf[:10],  # максимум 10 паттернов
        "skip_topics": context.get("skip_topics", []),
    }

    output = _user / ".compiled.json"  # в .gitignore
    output.write_text(
        json.dumps(compiled, ensure_ascii=False, separators=(",", ":")),
        encoding="utf-8"
    )
    return output
```

Результат — агент читает один файл в ~80–120 токенов вместо четырёх файлов в ~500–750:

```json
{"lang":"ru","style":"concise","autonomy":"high","expertise":"advanced",
"ides":["claude-code","codex","cursor"],"focus":"skills system design",
"active":["reefiki","skills-core"],"patterns":["предпочитает структуру до деталей (high)"],
"skip_topics":[]}
```

**Когда пересобирать:** git post-commit hook на изменения в `_user/`. Аналог `drift-cache.json`.

### Итоговая картина по токенам

```
Без оптимизации:
  Добавка _user/ к bootstrap: +500–750 токенов/сессию
  Через полгода (patterns разросся): +1000–1500 токенов/сессию

С оптимизацией (компактный формат + compiled profile):
  Добавка _user/ к bootstrap: +80–120 токенов/сессию
  Через полгода: +80–120 токенов/сессию (стабильно)

Итог: compiled profile делает _user/ практически бесплатным
```

### Порядок внедрения

Сначала разделение доменов в AGENTS.md (бесплатно, снимает конфликты). Потом компактный формат profile.md при создании wizard'ом. Compiled profile — когда `_user/patterns.md` реально начнёт расти, то есть через 2–4 недели активного использования. Раньше — преждевременная оптимизация.
