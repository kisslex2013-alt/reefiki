## 3) Тема 1 — Лицензирование и юридика

### Что есть сейчас

- В репозитории не обнаружен файл `LICENSE`.
- В `docs/` и основных аудитах эта тема либо отсутствует, либо упоминается вскользь.

### Почему это важно

Для open-source / public snapshot / потенциальной монетизации отсутствие лицензии — это не нейтральный дефолт, а юридическая неопределённость:

- внешние пользователи не знают, можно ли форкать, модифицировать и переиспользовать код;
- GitHub-репозиторий без лицензии по умолчанию считается “all rights reserved”; 
- это снижает доверие для интеграторов и соавторов;
- если проект планируется как продукт или как публичный knowledge-control-plane, лицензия влияет на:
  - коммерческое использование;
  - принятие PR от сообщества;
  - копирование шаблонов, prompt-логики, схем и документации;
  - права на контент, который создают агенты и пользователи.

### Риски

- юридическая неоднозначность для пользователей и контрибьюторов;
- блокировка внешнего adoption;
- риск несоответствия между публичной демонстрацией и реальными правами на использование;
- конфликт между private/public snapshot workflow и правами на содержащиеся данные.

### Рекомендации

1. Добавить `LICENSE` в корень репозитория.
2. Если цель — максимальное распространение, обычно рассматривают:
   - MIT / Apache-2.0 — для кода;
   - отдельную политику для контента и данных, если это нужно.
3. Прописать в `README` или отдельном `LEGAL.md`:
   - что лицензируется;
   - что не лицензируется;
   - как трактуются user-generated notes, agent-generated artifacts, public snapshots.
4. Если есть коммерческие намерения, отдельно зафиксировать:
   - можно ли использовать бренд;
   - можно ли использовать prompt-templates;
   - кто владеет сгенерированными материалами.

### Приоритет

**P0 / High** — не как кодовый баг, а как блокирующая product/legal неопределённость.

## 4) Тема 2 — Локализация и мультиязычность

### Что есть сейчас

- В репозитории есть русская и англоязычная документация.
- Есть визуальные материалы и русские версии ассетов.
- В коде и доках не найдено явной настройки FTS5-токенизации для русского языка.
- Поиск по `unicode61`/`cyrillic`/`migration` в основной кодовой части не дал признаков готовой локализационной настройки.

### Почему это важно

Для проекта, где контент содержит русский язык, вопрос не только в переводе UI. Ключевая проблема — качество поиска:

- FTS5 по умолчанию не решает морфологию русского языка;
- без правильной токенизации и нормализации поиск по словоформам может быть слабым;
- кириллица, дефисы, латиница, mixed-language заметки и translit-представления требуют отдельной модели индексации;
- если пользователь пишет по-русски, а индекс строится “английским образом”, появляются ложные пустые результаты.

### Риски

- поиск по русским словоформам может быть неудовлетворительным;
- смешанные RU/EN заметки будут индексироваться неравномерно;
- пользователь будет считать, что “знание потеряно”, хотя проблема только в tokenization/search strategy;
- ухудшается ценность всей базы знаний, потому что search — это основная точка доступа.

### Что нужно проверить технически

1. Как именно создаётся FTS5-индекс: используется ли стандартный tokenizer или кастомная настройка.
2. Как работают:
   - падежи и склонения;
   - множественное число;
   - знаки препинания;
   - латиница + кириллица в одном документе;
   - поиск по заголовкам и body.
3. Есть ли поведение “содержимое есть, но поиск его не находит”.
4. Нужен ли `unicode61`, `tokenchars`, `remove_diacritics`, внешняя нормализация или даже отдельный слой лемматизации.

### Рекомендации

- Добавить отдельный аудит/эксперимент по поиску на RU/EN корпусе.
- Подготовить набор тестовых документов:
  - только русский;
  - русский + английский;
  - с дефисами;
  - с числами и кодом;
  - с именами собственными.
- Сформировать критерий качества:
  - precision@k;
  - recall на наборе контрольных запросов;
  - время ответа;
  - число ложных пустых результатов.
- Если нужен действительно полезный русскоязычный поиск, рассмотреть:
  - нормализацию токенов до индексации;
  - морфологический слой поверх FTS;
  - отдельный search pipeline для RU.

### Приоритет

**P1 / High** — для русскоязычного knowledge-base это фундаментальное качество продукта.

## 5) Тема 3 — Производительность на реальных данных

### Что есть сейчас

- В аудитах обсуждаются архитектурные риски, но нет бенчмарков на объёмах вроде 500 / 1000 / 5000 страниц.
- Не найдено явной системы performance regression checks.
- Не найдено документации вида “сколько занимает rebuild index / search / harvest / publish при N документов”.

### Почему это важно

У REEFIKI индекс — это не второстепенный кеш. Он определяет UX:

- насколько быстро открывается поиск;
- насколько часто пользователи готовы запускать rebuild;
- не превращается ли `build_index()` в минутный/часовой bottleneck;
- не деградирует ли search при росте корпуса.

Без замеров проект легко оказывается в ловушке “на моей тестовой базе всё быстро”, а на реальном корпусе уже нет.

### Что нужно измерить

Минимальный бенчмарк-пакет:

- `build_index()`:
  - 500 страниц;
  - 1000 страниц;
  - 5000 страниц;
  - отдельно с короткими и длинными body.
- `search()`:
  - query latency p50/p95;
  - latency на пустом, среднем и большом корпусе;
  - latency при нескольких одновременных вызовах.
- `rebuild after missing/corrupt DB`:
  - cold rebuild;
  - warm rebuild;
  - влияние WAL.
- `publish / harvest`:
  - time-to-commit на реальном наборе файлов.

### Риски

- rebuild слишком медленный для интерактивного использования;
- индекс может быть приемлемым на 200 страницах и ломаться по UX на 5000;
- отсутствие benchmark-baseline делает невозможным сравнение после оптимизаций;
- нельзя отличить регрессию от нормального роста корпуса.

### Рекомендации

1. Добавить `bench/` или `tests/perf/` с синтетическим генератором корпуса.
2. Зафиксировать baseline в документации.
3. Завести performance-budget, например:
   - rebuild 1000 страниц < X секунд;
   - search p95 < Y мс;
   - health-check < Z мс.
4. Автоматизировать хотя бы один smoke benchmark в CI.

### Приоритет

**P1 / High** — особенно если проект предполагает регулярный рост knowledge base.

## 6) Тема 4 — Disaster Recovery и бэкап-стратегия

### Что есть сейчас

- Уже есть отдельный аудит по backup/recovery, но там фокус в основном на SQLite index, git history и memoir store.
- Не хватает именно полного сценария disaster recovery для конечного пользователя: “что делать, если сломалось всё”.

### Что недопокрыто

- повреждение не только `index.sqlite`, но и сопутствующих артефактов;
- потеря remote / неправильный force-push / конфликт зеркал;
- потеря working memory вне git;
- восстановление после частичного corruption в markdown-дереве;
- приоритеты восстановления: сначала данные, потом индекс, потом UI/интеграции.

### Почему это важно

Для knowledge-base система ценна не только в нормальном режиме. Самый дорогой момент — восстановление после аварии.

Если recovery процесс неполный, то:

- пользователи не доверяют сохранности;
- проект превращается в “хрупкий локальный инструмент”; 
- интеграции с мобильным capture и сторонними каналами становятся рискованными.

### Рекомендуемая DR-модель

**Уровень 1 — Source of truth**

- Markdown-страницы в git.
- Чёткая инструкция, как восстановить рабочее дерево из commit/remote.

**Уровень 2 — Rebuildable cache**

- SQLite индекс пересобирается.
- Должна быть команда/документ “index rebuild”.

**Уровень 3 — Ephemeral working memory**

- Memoir store и прочие временные данные должны иметь export/import или snapshot.

**Уровень 4 — External capture channel**

- Telegram / Pipedream / GitHub Actions цепочка должна иметь fallback, если внешний сервис недоступен.

### Рекомендации

- Добавить `RECOVERY.md` с пошаговыми сценариями:
  - потерян только индекс;
  - потерян `projects/<name>`;
  - повреждён git remote;
  - потерян memoir store;
  - неверно попал private content в public snapshot;
  - нужно сверить целостность после восстановления.
- Добавить “recovery checklist” в `docs/INDEX.md`.
- Зафиксировать RPO/RTO хотя бы в rough terms.

### Приоритет

**P0 / High** — если проект позиционируется как надёжный knowledge control plane.

## 7) Тема 5 — Accessibility и UX для нетехнических пользователей

### Что есть сейчас

- Документация и аудиты явно ориентированы на технического пользователя.
- Mobile capture есть, но сам workflow всё ещё требует понимания GitHub PAT, Pipedream, webhook/dispatch, репозитория и структур каталогов.

### Почему это важно

У REEFIKI есть потенциал как системы для:

- исследователей;
- менеджеров;
- гуманитариев;
- людей, которые не хотят думать в терминах “index.sqlite”, “FTS5”, “commit”, “dispatch”.

Если UX не адаптирован, то продукт остаётся tool-for-engineers, а не general-purpose PKM.

### Барьеры

- терминология в документации слишком инженерная;
- неочевидно, как новичку отличить inbox, wiki, plans, seen;
- нет понятного onboarding path;
- нет guided recovery;
- нет checklists “что делать дальше” после capture.

### Рекомендации

1. Сделать отдельную “non-technical quickstart” документацию.
2. Упростить словарь:
   - inbox = входящие;
   - wiki = знания;
   - index = поиск;
   - plan = задача/план;
   - seen = обработанное.
3. Добавить “happy path”:
   - capture;
   - review;
   - promote to wiki;
   - search;
   - recovery.
4. Обновить тексты ошибок так, чтобы они были полезны людям без инженерного бэкграунда.
5. Проверить, можно ли пройти основной сценарий без терминала вообще.

### Приоритет

**P2 / Medium** — не блокер для ядра, но критично для расширения аудитории.

## 8) Тема 6 — Mobile и capture-flow

### Что есть сейчас

В `scripts/mobile-capture/README.md` описан capture-flow через:

- Telegram bot;
- Pipedream;
- GitHub Actions dispatch;
- создание файлов в `inbox/`.

Это сильная идея, потому что user friction низкий: отправил ссылку — материал попал в систему.

### Что покрыто поверхностно

- безопасность токенов;
- надёжность внешнего pipeline;
- альтернативы Pipedream;
- повторная доставка и дедупликация;
- off-line или degraded-mode сценарии;
- контроль состояния: “дошёл ли URL до inbox?”.

### Риски

- Pipedream становится single point of failure;
- токен Telegram/GitHub PAT может быть скомпрометирован;
- double-send / duplicate capture;
- отсутствие идемпотентности;
- при сбое пользователь не понимает, дошла ли заметка.

### Рекомендации

- Ввести идемпотентный `capture_id`.
- Добавить retry policy и dead-letter / manual review path.
- Встроить подтверждение доставки:
  - accepted;
  - queued;
  - committed;
  - indexed.
- Рассмотреть альтернативы Pipedream:
  - GitHub Actions only;
  - self-hosted webhook relay;
  - n8n / Make / custom serverless endpoint.
- Документировать rotation и revocation для всех токенов.

### Приоритет

**P1 / High** — потому что capture-flow часто становится первым и самым важным пользовательским сценарием.

## 9) Тема 7 — Obsidian-интеграция

### Что есть сейчас

- Obsidian используется как viewer / внешний редактор по смыслу экосистемы.
- В документах нет полноценного аудита взаимодействия с `.obsidian/` и связанной конфигурацией.

### Почему это важно

Если Obsidian — часть workflow, появляются классические проблемы:

- одновременное редактирование одной и той же заметки через CLI и Obsidian;
- конфликт автосохранения;
- расхождение preview/markdown modes;
- плагины, меняющие фронтматтер или форматирование;
- риск того, что пользователь “видит одно”, а индекс строится по другому.

### Риски

- конфликт версий и silent overwrite;
- порча frontmatter из-за плагинов/автоформатирования;
- несогласованность индекса и визуального представления;
- лишняя сложность для non-technical user.

### Рекомендации

1. Определить Obsidian как:
   - read-only viewer;
   - primary editor;
   - or optional companion.
2. Явно задокументировать rules of engagement:
   - что можно редактировать в Obsidian;
   - что лучше не трогать вручную;
   - как обновлять индекс после редактирования.
3. Проверить совместимость с `.obsidian/` и плагинами.
4. Если нужен двусторонний workflow, добавить:
   - conflict detection;
   - change timestamps;
   - safe reindex after edits.

### Приоритет

**P2 / Medium** — важная UX-тема, особенно если Obsidian — один из основных интерфейсов.

## 10) Тема 8 — Эволюция схемы и migrations

### Что есть сейчас

- Архитектура уже содержит структурированный индекс и YAML/frontmatter-поля.
- Но нет полноценного, явно документированного migration layer для массового изменения схемы.

### Почему это важно

Как только у вас есть:

- 200+ страниц;
- несколько типов документов;
- индекс, который зависит от структуры frontmatter;
- новые обязательные поля;

…любая эволюция схемы становится миграцией данных, а не просто “добавим поле”.

### Типовые проблемы

- старые страницы не имеют нового обязательного поля;
- часть документов заполнена вручную, часть агентами;
- нужно одновременно поддерживать old schema и new schema;
- backfill может ломать ссылки, search ranking или display logic;
- если миграция не идемпотентна, повторный запуск портит данные.

### Рекомендации

- Ввести versioned schema contract.
- Для каждого schema change писать:
  - what changes;
  - how to migrate;
  - how to backfill;
  - how to validate.
- Добавить migration tests на реальном корпусе или на фикстурах.
- Предусмотреть “dry run” режим для миграций.
- Хранить миграционную историю отдельно от основной документации.

### Приоритет

**P1 / High** — особенно если roadmap включает новые обязательные поля и более строгую структуру.

## 11) Тема 9 — Конкурентный анализ

### Что есть сейчас

В стратегии и аудитах уже есть сравнения с частью PKM/knowledge tools, но не хватает более широкого рынка.

### Что полезно добавить

Сравнение с:

- Notion AI;
- Roam Research;
- Logseq;
- Capacities;
- Mem0 / Zep;
- Obsidian ecosystem;
- возможно, AI-native note apps и knowledge layers.

### Зачем это нужно

Без конкурентного анализа легко строить продукт “в вакууме”:

- технически сильный, но без чёткой дифференциации;
- с плохим onboarding, потому что команда не знает, с чем реально сравнивает UX;
- без понимания, где REEFIKI выигрывает:
  - агент-агностичность;
  - локальный control plane;
  - Markdown source of truth;
  - multi-vendor memory governance;
  - workflow portability.

### Что должен отвечать конкурентный анализ

- чем REEFIKI принципиально лучше;
- где он слабее;
- кому он нужен, а кому нет;
- какие 3–5 killer features должны быть безупречны;
- где не стоит догонять рынок, а стоит делать иначе.

### Рекомендации

- Сделать отдельный market map:
  - consumer PKM;
  - team knowledge bases;
  - AI memory layers;
  - local-first note apps;
  - automation-first workflows.
- Сформировать messaging matrix:
  - what it is;
  - what it is not;
  - who it is for.
- Обновлять сравнение раз в несколько месяцев.

### Приоритет

**P2 / Medium** — важный стратегический слой, но не срочный блокер для core system.

## 12) Тема 10 — Метрики здоровья базы знаний

### Что есть сейчас

- Нет явного health dashboard для качества знаний.
- Не найдено набора KPI для оценки того, работает ли дистилляция.

### Почему это важно

Knowledge base может “работать” технически, но деградировать смыслово:

- страницы копятся, но не используются;
- дубликаты растут;
- индекс полон шумовых заметок;
- пользователи не могут найти нужное;
- дистилляция превращается в склад.

### Какие метрики нужны

Минимум:

- **coverage** — доля источников, прошедших обработку;
- **freshness** — как давно обновлялись ключевые страницы;
- **usefulness / use count** — как часто страницы реально используются;
- **search success rate** — сколько запросов приводят к полезному результату;
- **dedup ratio** — сколько дублей удаётся отбрасывать;
- **distillation yield** — сколько сырых материалов превращается в полезные страницы;
- **staleness** — сколько страниц устарело;
- **broken links / schema errors**;
- **capture-to-publish lag** — сколько времени от входящего материала до полезного knowledge artifact.

### Почему без этого плохо

- невозможно понять, помогает ли система или просто хранит мусор;
- трудно приоритизировать рефакторинг;
- невозможно сравнивать версии алгоритмов;
- нельзя доказать пользу knowledge operations.

### Рекомендации

1. Добавить `health`/`metrics` отчёт.
2. Ввести dashboard по ключевым KPI.
3. Определить “healthy knowledge base” как набор числовых порогов.
4. Сохранить history metrics, чтобы видеть тренды.
5. Связать метрики с действиями:
   - если staleness растёт — нужна ревизия;
   - если search success падает — проблема в index/search;
   - если dedup ratio низкий — проблема в capture normalization.

### Приоритет

**P0 / High** — это фундамент управления качеством, а не просто nice-to-have.

## 14) Практический вывод

Эти темы не выглядят как отдельные баги. Это скорее недостающие “системные слои”, без которых REEFIKI остаётся сильным, но неполным инженерным инструментом.

Если коротко:

- **без лицензии** проект юридически неясен;
- **без локализации** он хуже работает для RU-корпуса;
- **без бенчмарков** нельзя понять масштабируемость;
- **без DR** нельзя доверять хранению знаний;
- **без UX для non-tech** он не выйдет за пределы инженеров;
- **без mobile-flow hardening** capture остаётся хрупким;
- **без migration layer** схема будет ломать старые данные;
- **без конкурентного анализа** непонятно позиционирование;
- **без metrics** невозможно управлять качеством базы знаний.

Это хороший список для следующей волны работ после закрытия уже найденных security/concurrency/quality issues.
