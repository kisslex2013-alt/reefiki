# OpenViking → REEFIKI: переработанный анализ

> Источник: реальный репо `kisslex2013-alt/reefiki` (прочитаны README, AGENTS.md, ROADMAP.md)

---

## Контекст: что такое REEFIKI на самом деле

REEFIKI — это **local-first, markdown-native, agent-agnostic вики-шаблон** для персональной базы знаний. Никаких рифов, никакой морской биологии. Стек: markdown + SQLite FTS5 + git + `scripts/reefiki.py`. Философия: дистилляция, не архивация. Инфраструктура намеренно минимальна — каждый новый слой подключается только по объективному триггеру.

OpenViking, напротив, — это **тяжёлая серверная инфраструктура**: Python-сервер + Rust CLI + C++ core extensions + embedding-модель + VLM-модель + векторный индекс. Лицензия — AGPL-3.0.

**Вывод сразу:** брать код из OpenViking нельзя (AGPL) и не нужно (несовместимый стек). Взять можно только **архитектурные паттерны** — те идеи, которые OpenViking реализовал кодом, а REEFIKI может реализовать **конвенциями, frontmatter-полями и изменениями в command specs**.

---

## Что REEFIKI уже делает из того, что есть в OpenViking

Прежде чем говорить «брать», нужно зафиксировать **что уже реализовано**, чтобы не изобретать колесо:

| Концепция OpenViking | Статус в REEFIKI |
|---|---|
| Filesystem paradigm | ✅ **Уже есть** — `projects/<name>/wiki/<subdir>/` — это и есть файловая иерархия как парадигма |
| Unified context types (resources/user/agent) | ✅ **Частично** — `memoir` / `REEFIKI` / `graphify` — ровно те же три роли |
| Observable retrieval (provenance) | ✅ **Частично** — `/query` с provenance уже реализован в Phase 0g |
| Session management → `/harvest` | ✅ **Есть**, но review-first, а не auto-capture — это архитектурный выбор, не упущение |
| Skills isolation + cross-project sharing | ✅ **Есть** — `wiki/skills/` + cross-project search из диспетчерского режима |

## Идея 1 — Трёхуровневое представление страниц (L0 / L1 / L2)

### Что это в OpenViking

OpenViking автоматически создаёт три представления каждого контента:
- **L0 (Abstract)** — ≈1 предложение, для быстрой проверки релевантности
- **L1 (Overview)** — ≈2 000 токенов, для принятия решений о чтении
- **L2 (Details)** — полный оригинал, читается только по необходимости

Агент начинает с L0, принимает решение, поднимается до L1, и только потом — L2.

### Почему это актуально для REEFIKI

В REEFIKI сейчас `wiki/index.md` частично выполняет роль L0/L1 — но **только для индекса, не для содержимого страниц**. Когда агент делает `/query`, он читает index.md целиком (L1), затем загружает нужные страницы полностью (L2). **L0-фильтрации нет**: агент не может быстро проверить, релевантна ли страница, не загрузив её.

При 100+ страницах это начинает создавать контекстный шум. ROADMAP уже упоминает «progressive disclosure» как направление, но механизм не зафиксирован.

### Что конкретно взять

**1. Добавить поле `abstract` в `_schema.md`** — обязательное однострочное резюме страницы (≤120 символов).

```yaml
---
id: auth-jwt-decision-001
type: decision
title: Использовать JWT для аутентификации
abstract: "Решение в пользу JWT вместо сессий: причины, компромиссы, дата пересмотра."
useful_when: "при архитектурном ревью или выборе auth-подхода в новом сервисе"
...
---
```

**2. Обновить `wiki/index.md` template** — включить `abstract` в каждую запись индекса (рядом с `useful_when`).

**3. Обновить `.claude/commands/query.md`** — добавить явный L0-шаг перед загрузкой страниц:

```
Шаг 1. Прочитать только поле `abstract` из записей index.md — без загрузки страниц.
Шаг 2. Отобрать ≤5 кандидатов по совпадению с запросом.
Шаг 3. Загрузить только эти страницы целиком (L2).
Шаг 4. Синтезировать ответ с provenance.
```

**Где менять:** `projects/_template/wiki/_schema.md`, `projects/_template/wiki/index.md`, `projects/_template/.claude/commands/query.md`

**Экономия:** при 80 страницах — агент читает `abstract` всех (≈80 × 120 символов ≈ 2 400 токенов) вместо загрузки полного index.md (≈10 000+ токенов). Нужные страницы — только 3–5 из 80.

## Идея 2 — Directory-level Overview (`_overview.md`)

### Что это в OpenViking

У каждой директории есть `.overview`-файл — структурированное L1-описание содержимого раздела: что здесь лежит, когда смотреть сюда, ключевые связи.

### Почему это актуально для REEFIKI

Когда `/query` ищет ответ в большой вики, он сейчас работает плоско: FTS5 по всем страницам сразу. Нет механизма «сначала понять, в каком разделе искать». По мере роста wiki (`decisions/`, `concepts/`, `skills/`, `synthesis/`, `entities/`) плоский поиск будет давать шум.

ROADMAP признаёт эту проблему и резервирует Phase 1 (qmd) под локальный семантический поиск. Но **до qmd** можно улучшить навигацию через явные directory summaries — без новых зависимостей.

### Что конкретно взять

**1. Добавить `_overview.md` в каждый `wiki/<subdir>/`** в `_template`:

```
wiki/
├── decisions/_overview.md    # "Архитектурные решения проекта. Смотри сюда при выборе..."
├── concepts/_overview.md     # "Ключевые понятия домена. Смотри при onboarding-е..."
├── skills/_overview.md       # "Проверенные процедуры. Смотри при повторяющихся задачах..."
├── synthesis/_overview.md    # "Выводы из исследований. Смотри при..."
```

Формат простой: 3–5 предложений о том, что лежит в разделе, когда сюда смотреть, и 3–5 ключевых ссылок на самые важные страницы раздела.

**2. Обновить `/query` — добавить directory routing**:

```
Шаг 1. По запросу — определить, в каком разделе (decisions/concepts/skills/synthesis) вероятнее ответ.
Шаг 2. Прочитать `_overview.md` этого раздела (L1).
Шаг 3. Если нужно — FTS5 внутри раздела + abstract-фильтр (Идея 1).
Шаг 4. Загрузить конкретные страницы (L2).
```

**Где менять:** добавить `_overview.md` в `projects/_template/wiki/*/`, обновить команду `/query`.

**Стоимость:** 0. Только конвенция и файлы. Полностью обратимо.

## Идея 3 — Явное разделение User Memory и Agent Memory

### Что это в OpenViking

OpenViking делит память на три чётких контейнера:
- `user/memories/` — предпочтения пользователя (стиль общения, рабочие паттерны)
- `agent/memories/` — опыт агента (что работало, что нет)
- `resources/` — внешние знания (документы, репо, веб)

### Почему это актуально для REEFIKI

REEFIKI уже имеет трёхслойную архитектуру `memoir` / `REEFIKI` / `graphify`, которая семантически совпадает с OpenViking. Однако **`memoir`-слой (short working memory + preferences) не имеет формализованной схемы** для user/agent разделения. AGENTS.md упоминает `_user/profile.md`, `_user/context.md`, `_user/patterns.md` в контексте сессионных данных — но это не зафиксировано в `_template`.

### Что конкретно взять

**1. Добавить `_user/` директорию в `_template`** с явным разделением:

```
projects/_template/
├── _user/
│   ├── profile.md     # кто пользователь, его expertise, стиль работы
│   ├── patterns.md    # устойчивые паттерны (как он обычно формулирует, что ему важно)
│   └── log.md         # дельта-обновления (append-only, как wiki/log.md)
```

**2. Обновить `/harvest`** — добавить опциональный шаг: после дистилляции wiki-знаний — отдельно зафиксировать, изменилось ли что-то в `_user/patterns.md` (как пользователь работал в этой сессии, что предпочитал).

**3. Добавить в AGENTS.md rule**: при `/query` — сначала проверить `_user/patterns.md` на релевантные предпочтения и адаптировать ответ. (Пример: пользователь предпочитает code-first — показать код, потом объяснение.)

**Где менять:** добавить `_user/` в `projects/_template/`, обновить корневой AGENTS.md (правило чтения `_user/` в начале сессии), обновить `projects/_template/AGENTS.md`.

## Идея 4 — Confidence-level tagging для cross-reference edges

### Что это в OpenViking

OpenViking не отслеживает источники ссылок между контент-узлами. Это упущение, которое ROADMAP REEFIKI уже заметил самостоятельно.

Но **ROADMAP уже упоминает**: «confidence tagging для связей между страницами (по образцу graphify): `EXTRACTED` / `INFERRED` / `AMBIGUOUS`» — как опцию при активации Phase 1.

### Почему это актуально для REEFIKI раньше Phase 1

OpenViking показывает, что **retrieval trajectory** (по какому пути агент нашёл ответ) — это самостоятельная ценность: пользователь видит, откуда взялся ответ, и может исправить плохой источник.

В REEFIKI `/query` уже возвращает provenance (phase 0g). Но это только список страниц. Нет ответа на «почему именно эти страницы» — не видна навигационная цепочка.

### Что конкретно взять

**1. Добавить `confidence` в `_schema.md`** для wikilinks:

```markdown
# В теле страницы — явный уровень уверенности при ссылках:
Решение связано с [[auth-jwt-decision-001]] (EXTRACTED — прямо цитируется в источнике)
и вероятно повлияет на [[rate-limiting-concept-002]] (INFERRED — не явно, но следует из архитектуры)
```

**2. Обновить вывод `/query`** — показывать не только список страниц, но и «цепочку»:

```
Ответ основан на:
  → decisions/auth-jwt-decision-001.md (найдено через L0-abstract → confirmed L2)
    ↳ связано с: concepts/session-vs-token.md (EXTRACTED)
    ↳ может влиять: skills/token-refresh.md (INFERRED)
```

**Где менять:** `projects/_template/wiki/_schema.md` (добавить поле `related` с confidence), `projects/_template/.claude/commands/query.md` (обновить формат вывода).

## Что НЕ брать из OpenViking

| Компонент | Почему не нужно |
|-----------|-----------------|
| **Сервер + VLM + embedding** | Противоречит local-first философии REEFIKI; триггер для vectors — только доказанные FTS5 misses (Phase 1) |
| **L2 (full content) как отдельный файл** | В REEFIKI L2 — это сама wiki-страница. Не нужен дополнительный уровень |
| **Auto-evolving memory без review** | REEFIKI сознательно review-first. Auto-capture = memory spam — это зафиксировано в ROADMAP |
| **`viking://` URI-протокол** | REEFIKI уже имеет файловые пути как адресацию; отдельный протокол — overhead без выигрыша |
| **Код (AGPL-3.0)** | Лицензионная несовместимость с Apache 2.0 REEFIKI |

## Главный вывод

OpenViking решает серверной инфраструктурой ровно те же проблемы, которые REEFIKI решает конвенциями и контрактами. **Это подтверждение правильности архитектурного выбора REEFIKI**, а не указание на недостатки.

Разрыв только один: **прогрессивная загрузка контекста (L0/L1/L2)** в REEFIKI сейчас реализована частично (index.md как L1), но не на уровне отдельных страниц и не зафиксирована в `_schema.md` и command specs как явная конвенция.

## Приоритеты внедрения

```
НЕМЕДЛЕННО (только frontmatter + конвенция, 0 зависимостей):
  1. Поле `abstract` в _schema.md и index.md
  2. _overview.md в wiki/<subdir>/ в _template
  3. Обновление query.md — L0-шаг перед загрузкой страниц

СЛЕДУЮЩИЙ ШАГ (когда wiki выросла до 50+ страниц):
  4. _user/ директория в _template
  5. Confidence tags для wikilinks в _schema.md

ПО TRIГГЕРУ Phase 1 (100+ страниц / FTS5 misses):
  → qmd + идея Directory Recursive Retrieval становится полноценно применима
     (OpenViking подтверждает, что иерархический поиск нужен при масштабе)
```
