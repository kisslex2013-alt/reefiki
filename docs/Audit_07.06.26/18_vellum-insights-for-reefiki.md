# Идеи из Vellum Assistant для развития REEFIKI

> Источник: [github.com/vellum-ai/vellum-assistant](https://github.com/vellum-ai/vellum-assistant)  
> Vellum — персональный AI-ассистент с памятью, личностью и проактивными уведомлениями.  
> Всё ниже — архитектурные паттерны, адаптированные под markdown-природу REEFIKI без TypeScript-инфраструктуры.

---

## 1. Эфемерный скрэтчпад сессии — `NOW.md`

**Откуда:** В Vellum есть `NOW.md` — живой файл текущего фокуса и активных тредов. Он эфемерный: не архивируется, перезаписывается каждую сессию.

**Что взять в REEFIKI:**  
Добавить `NOW.md` в корень проекта. Агент ведёт его сам в течение сессии и обнуляет при старте новой.

```markdown
# NOW — текущая сессия

## Задача
[что делаем прямо сейчас]

## Уже решено сегодня
- [X] реализована авторизация через JWT
- [X] отказались от cookie-based — причина в decisions/auth-2024.md

## Активные ограничения
- не трогать auth-модуль
- структура БД заморожена до v2.0

## Следующий шаг
[что будем делать]
```

**Почему важно:** При смене сессии `NOW.md` становится основой хэндофф-промпта автоматически. Это закрывает проблему авто-компакта без дополнительных команд.

---

## 2. Окна устаревания по типу записи

**Откуда:** В Vellum каждый тип памяти имеет свой `staleness window` — факты об идентичности живут 6 месяцев, события — 3 дня. Старые записи не участвуют в поиске и не инжектируются в контекст.

**Что взять в REEFIKI:**  
Добавить поле `expires` в шаблоны каждого типа записи.

| Тип вики | Рекомендуемый TTL |
|---|---|
| `decisions/` | 6 месяцев (архитектурные решения) |
| `skills/` | 3 месяца (техники устаревают) |
| `synthesis/` | 2 недели (выводы сессий) |
| `concepts/` | 12 месяцев (фундаментальные понятия) |

```markdown
---
type: decision
created: 2025-01-15
expires: 2025-07-15
useful_when: выбор между state-менеджерами
---

# Отказались от Redux

**Решение:** Zustand  
**Почему не Redux:** избыточный бойлерплейт для нашего масштаба  
**Почему не MobX:** команда не знает, высокий порог входа
```

**Почему важно:** Без TTL вики превращается в кладбище. Устаревшие решения продолжают влиять на контекст и создают «решение, принятое призраком».

## 3. Принцип интеллектуальной честности агента

**Откуда:** В `AGENTS.md` у Vellum прямо прописано: «Defend your technical positions. If you change your mind, explain what new information changed it — not just that the user questioned it. Sycophantic responses erode trust.»

**Что взять в REEFIKI:**  
Добавить этот принцип в `AGENTS.md` проекта.

```markdown
## Интеллектуальная честность

Отстаивай технические позиции. Если меняешь мнение — объясни,
какая новая информация изменила его, а не просто согласись с пользователем.
Соглашательство разрушает доверие и ухудшает результат.

Перед сохранением решения в wiki/decisions/ убедись:
- оно действительно принято, а не продиктовано давлением
- причина отказа от альтернатив зафиксирована честно
```

**Почему важно:** Агент, который соглашается на давление, будет сохранять «решения» в вики, которых на самом деле не было. Это отравляет базу знаний.

## 4. Минимализм системного промпта

**Откуда:** В Vellum жёсткое правило: «Adding content to the system prompt is a last resort. Every token added increases latency, cost, and crowds out user context.» Паттерн: Skill first → Config second → Code third → System prompt only if nothing else works.

**Что взять в REEFIKI:**  
Явно прописать в `AGENTS.md` иерархию инжекции контекста при старте сессии.

```markdown
## Иерархия загрузки контекста (от приоритетного к менее)

1. NOW.md — текущая задача и активные ограничения (всегда)
2. wiki/decisions/ — только те, что relevant к текущей задаче
3. wiki/skills/ — только те, что useful_when совпадает с задачей
4. wiki/synthesis/ — только последние 2 записи
5. _domain.md — только если сессия первая или задача выходит за рамки проекта

Не грузи всё сразу. Чем меньше контекста — тем чище фокус.
```

**Почему важно:** REEFIKI сейчас не регулирует что и когда инжектируется. Без этого `/query` рискует тащить всю вики в контекст.

---

## 5. Проактивность агента — самодиагностика сессии

**Откуда:** В Vellum есть Proactivity Engine — каждый час агент перечитывает свои заметки, замечает незавершённое и выходит на связь первым.

**Что взять в REEFIKI:**  
Не внешний таймер (это инфраструктура), а поведенческий триггер в `AGENTS.md`.

```markdown
## Самодиагностика сессии

Проверяй признаки деградации контекста и предлагай /checkpoint:

Сигналы деградации:
- Повторяешь вопрос, который уже задавал в этой сессии
- Предлагаешь решение, которое уже было отвергнуто (проверь decisions/)
- Ходишь по кругу: одна и та же проблема возникает третий раз
- Видишь, что NOW.md устарел больше чем на 30% от текущего состояния

При обнаружении → скажи пользователю:
"Замечаю признаки перегрузки контекста. Рекомендую /checkpoint — займёт минуту."
```

**Почему важно:** Это превращает REEFIKI из реактивного в проактивный инструмент без инфраструктурных зависимостей.

---

## 6. Манифест-ориентированные навыки

**Откуда:** В Vellum скиллы — это `SKILL.md + TOOLS.json`. Манифест описывает что умеет навык, когда он активируется, какие инструменты инжектирует. Скиллы загружаются по требованию, не всегда.

**Что взять в REEFIKI:**  
Обогатить шаблон `wiki/skills/` метаданными, чтобы навык был не просто текстом, а инструкцией к действию.

```markdown
---
type: skill
created: 2025-03-10
expires: 2025-06-10
useful_when: отладка race conditions в async/await коде
activation_signal: "race condition", "async bug", "Promise hell"
---

# Дебаггинг race conditions

## Симптомы
[как понять, что это именно race condition]

## Алгоритм
1. [шаг 1]
2. [шаг 2]

## Антипаттерны
- [что НЕ делать и почему]

## Проверенный сниппет
```ts
// код, который реально работает
```
```

Поле `activation_signal` — ключевые слова, при которых агент сам предлагает применить навык.

**Почему важно:** Сейчас skills в REEFIKI — пассивное хранилище. Манифест делает их активными.

---

## 7. Журнал рефлексии — `JOURNAL.md`

**Откуда:** В Vellum есть per-user journal — «ассистент записывает собственные рефлексии о прошедших взаимодействиях».

**Что взять в REEFIKI:**  
Добавить `wiki/journal.md` — агент пишет в конце сессии краткую рефлексию о качестве работы.

```markdown
# Журнал сессий

## 2025-05-20
- Задача: рефактор auth-модуля
- Что сработало хорошо: разбили на атомарные PR, каждый с decision
- Что пошло не так: не зафиксировали причину отказа от session tokens сразу, потом пришлось восстанавливать
- Что улучшить: в следующий раз сначала /save решение, потом код

## 2025-05-18
...
```

**Почему важно:** Вики хранит знания о проекте, журнал хранит знания о том, как с вами работает агент. Это мета-уровень, которого нет в REEFIKI.

## 8. Политика удаления мёртвых записей

**Откуда:** В Vellum правило «Dead Code Removal»: «Proactively remove unused code during every change. Ask: is there any code that nothing calls? If yes, delete it.»

**Что взять в REEFIKI:**  
Аналогичное правило для вики-записей — добавить в команду `/lint`.

```markdown
## /lint — расширенные проверки

Дополнительно к текущим проверкам:

Мёртвые записи:
- decisions/, skills/ записи с истёкшим expires — предложить удалить или обновить
- concepts/ без единой ссылки из других страниц — пометить как orphan
- synthesis/ старше 30 дней — предложить архивировать в seen/

Правило: вики должна быть живой, а не архивом.
Лучше 20 актуальных записей, чем 200 устаревших.
```

**Почему важно:** Без политики удаления вики растёт, `/query` начинает возвращать мусор, а агент тащит устаревший контекст.

---

## 9. Атрибуция источника в записях

**Откуда:** В Vellum память извлекается «с атрибуцией источника» — каждый memory item знает, откуда пришёл.

**Что взять в REEFIKI:**  
Добавить поле `source` в frontmatter записей.

```markdown
---
type: decision
source: session-2025-05-15 / PR #234
created: 2025-05-15
---
```

**Почему важно:** Когда решение вызывает вопросы, можно найти оригинальную сессию или PR и восстановить полный контекст. Без атрибуции записи становятся анонимными артефактами.

---

## 10. Стратегия восстановления при переполнении контекста

**Откуда:** В Vellum есть «Context overflow resilience» — детерминированный pipeline восстановления: preflight budget check → forced compaction → tool-result truncation → media stubbing → injection downgrade. Каждый уровень пробуется последовательно.

**Что взять в REEFIKI:**  
Добавить `overflow-recovery.md` — явный протокол для пользователя и агента.

```markdown
# Протокол восстановления при переполнении контекста

## Уровень 1 — Превентивный (70% контекста)
- Агент замечает заполнение и предлагает /checkpoint
- Сохраняем NOW.md, обновляем decisions если есть новые

## Уровень 2 — Тактический (85% контекста)
- Открываем чистую сессию
- Инжектируем только NOW.md + релевантные decisions (не всю вики)
- Продолжаем с минимальным контекстом

## Уровень 3 — Экстренный (контекст уже переполнен)
- Не ждём авто-компакта
- Быстро сохраняем текущий код-сниппет в inbox/
- Пишем минимальный хэндофф: "сделано X, нужно Y, стек Z, ограничения W"
- Новая сессия

## Что НИКОГДА не делать
- Не доверять авто-компакту
- Не начинать новую сессию без NOW.md
```

**Почему важно:** Сейчас у REEFIKI нет явного протокола что делать когда контекст уже переполнен. Этот файл даёт пошаговый ответ.
