# Как сделать REEFIKI проактивным: безопасно и по-настоящему

> Принцип: агент **делает** → сообщает. Не «предлагаю подумать», не «хотите оформить?».  
> Всё работает через текст в AGENTS.md — агностично к IDE и LLM.

---

## Что значит «безопасная проактивность»

Три критерия:

1. **Действие обратимо** — агент может сделать сам, потому что это легко откатить (создать файл, дополнить запись). Необратимые вещи (удалить, отправить, задеплоить) — только с явного подтверждения.
2. **Действие конкретно** — не «я бы рекомендовал сохранить», а просто сохраняет и говорит «сохранил в decisions/auth.md».
3. **Действие уместно** — триггер привязан к тому, что происходит прямо сейчас, а не к расписанию или общим ощущениям.

## 1. Проактивность при управлении контекстом

### Что делает агент сам

**При старте сессии** — читает `NOW.md` и сразу пишет одной строкой:
```
📍 Контекст загружен: задача «X», активных ограничений: 3, последнее решение — auth.md (2 дня назад).
```
Не спрашивает «хотите загрузить контекст». Просто делает и сообщает.

**При обнаружении деградации** — не предлагает «подумать о переходе», а прямо пишет:
```
⚠️ Замечаю: третий раз обсуждаем одно и то же. Выполняю /checkpoint.
[сохраняет NOW.md, пишет хэндофф-промпт]
Готово. Можешь открыть новую сессию и вставить это:
---
[текст хэндофф-промпта]
---
```

**Триггеры деградации (конкретные, не размытые):**
- Тот же вопрос задан дважды в сессии → автоматический checkpoint
- Агент предложил решение, которое уже есть в `decisions/` как отвергнутое → стоп + сообщение
- Пользователь написал «погоди», «стоп», «ты уже это предлагал» → checkpoint без вопросов

### Правило для AGENTS.md
```markdown
## Управление контекстом — автономное поведение

При старте: прочитай NOW.md, выведи одну строку состояния.
При деградации: не предлагай checkpoint — выполни его.
Checkpoint = обновить NOW.md + сгенерировать хэндофф-промпт + вывести его.
Хэндофф-промпт выводи в блоке кода, чтобы его было легко скопировать.
```

## 2. Проактивность при сохранении знаний

### Проблема текущего REEFIKI
Агент спрашивает «сохранить в вики?» — это перекладывает решение на пользователя.  
Пользователь устал, говорит «потом» — знание теряется.

### Что делает агент сам

**Паттерн «тихое сохранение»** — агент сохраняет сам, сообщает после:
```
[решает задачу]
...
💾 Зафиксировал в wiki/decisions/db-migration-strategy.md — ты отклонил партиционирование 
   в пользу индексов. Причина записана. Можешь проверить или удалить.
```

**Триггеры для тихого сохранения:**
- Прозвучало «мы решили», «договорились», «будем делать так» → сохраняет в `decisions/`
- Починили нетривиальный баг (больше 3 итераций) → сохраняет в `skills/`
- Пользователь написал «запомни» или «важно» → `inbox/` немедленно
- Конец длинной сессии (много обменов) → автоматически обновляет `NOW.md` и предлагает `synthesis/`

**Чего НЕ делает:**
- Не сохраняет каждое тривиальное решение (форматирование кода, именование переменных)
- Не спрашивает «это важно?» — сам оценивает по объёму обсуждения

### Правило для AGENTS.md
```markdown
## Тихое сохранение

Сохраняй без вопроса если:
- прозвучало «решили / договорились / будем делать так»
- баг исправлен после 3+ итераций
- пользователь написал «запомни» или «важно»

После сохранения — одна строка: «💾 Сохранил в [путь] — [одна фраза о содержимом]».
Не спрашивай разрешения заранее. Файл легко удалить.
```

## 3. Проактивность при обнаружении конфликтов

### Что делает агент сам

Перед тем как что-то предложить — сам проверяет вики на конфликты:

```
Ты просишь добавить кэширование через Redis.

⚡ Проверил decisions/ — нашёл конфликт:
   decisions/infra-2025-03.md: «Redis не используем — лицензия SSPL несовместима 
   с нашей лицензией MIT».

Варианты без Redis: Valkey (форк MIT), Dragonfly, in-memory через Map.
Какой предпочесть?
```

Агент не просто выполняет запрос — он сам находит противоречие и сразу даёт альтернативы.

**Триггеры проверки конфликтов:**
- Любое предложение нового инструмента/библиотеки → проверить `decisions/`
- Любое изменение архитектурного паттерна → проверить `decisions/`
- Новый подход к задаче → проверить `skills/` на существующий проверенный метод

### Правило для AGENTS.md
```markdown
## Проверка конфликтов

Перед любым архитектурным предложением:
1. Проверь wiki/decisions/ на слово из запроса
2. Если нашёл — выведи конфликт и сразу предложи альтернативы
3. Не жди, пока пользователь сам вспомнит о решении
```

## 4. Проактивность при качестве вики

### Что делает агент сам

**При открытии проекта раз в N сессий** — не просит «проверить вики», а сам делает аудит и выводит конкретный список:

```
🔍 Аудит вики (раз в 10 сессий):

Устарело (expires прошёл):
- decisions/eslint-config.md — истёк 2025-01-10. Удалить? [y/n]
- skills/webpack-optimization.md — истёк 2025-02-01. Актуален ещё?

Orphan-записи (никто не ссылается):
- concepts/event-sourcing.md — создана 4 месяца назад, не использовалась

NOW.md последний раз обновлялся 5 сессий назад — перезаписать?
```

Пользователь отвечает `y/n` — агент выполняет немедленно.

**Триггер:** счётчик сессий в `_meta.md` (инкрементируется при каждом старте).

### Правило для AGENTS.md
```markdown
## Аудит вики

Каждые 10 сессий (счётчик в _meta.md):
1. Найди записи с истёкшим expires
2. Найди orphan-записи без входящих ссылок
3. Выведи список конкретных файлов с вопросом [y/n]
4. Жди ответа — выполняй немедленно

Не делай аудит посреди задачи. Только при старте сессии.
```

## 5. Проактивность при смене задачи

### Что делает агент сам

Когда пользователь резко меняет тему — агент замечает это и обновляет `NOW.md` без вопроса:

```
[был разговор про авторизацию, пользователь вдруг пишет про оптимизацию запросов]

📍 Задача изменилась. Обновил NOW.md:
   Было: «рефактор auth-модуля»
   Стало: «оптимизация запросов к users-таблице»
   Предыдущий прогресс по auth сохранён в synthesis/auth-session-2025-06-06.md
```

**Триггер:** тема сообщения пользователя семантически далека от текущей задачи в `NOW.md`.

---

## 6. Что должно остаться реактивным

Проактивность не означает «делать всё». Некоторые вещи должны оставаться только по запросу:

| Действие | Почему только по запросу |
|---|---|
| Удаление записей вики | Необратимо — только после явного `y` |
| Реструктуризация вики | Слишком субъективно |
| Переименование файлов | Может сломать ссылки |
| Запись в `raw/` архив | Пользователь сам решает что архивировать |
| Создание нового проекта | Высокий риск ошибки |

---

## Итого: что добавить в AGENTS.md прямо сейчас

Пять правил, которые переводят REEFIKI в проактивный режим без инфраструктуры:

```markdown
## Проактивное поведение

### 1. Старт сессии
Всегда читай NOW.md. Выведи одну строку: задача + ограничения + последнее решение.
Не спрашивай — просто делай.

### 2. Деградация контекста
Триггеры → автоматический checkpoint без вопроса:
- тот же вопрос задан дважды
- предложил решение из rejected_alternatives
- пользователь написал «ты уже это предлагал»

### 3. Тихое сохранение
«решили / договорились / запомни» → сохрани в decisions/ или inbox/.
Сообщи одной строкой после. Не спрашивай до.

### 4. Проверка конфликтов
Перед архитектурным предложением → проверь decisions/.
Нашёл конфликт → покажи его + сразу дай альтернативы.

### 5. Аудит вики
Каждые 10 сессий (счётчик в _meta.md) → список устаревших и orphan-записей.
Только при старте сессии. Только конкретные файлы. Жди y/n.
```

Всё это работает в Codex, Claude Code, Cursor, Windsurf — любой агент читает `AGENTS.md`.
