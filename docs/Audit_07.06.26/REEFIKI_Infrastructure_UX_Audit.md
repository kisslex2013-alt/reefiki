# Дополнение к аудиту REEFIKI: Инфраструктура, UX и Экосистема (Слепые зоны)

Этот документ покрывает критические аспекты развития базы знаний REEFIKI, которые не вошли в базовые аудиты архитектуры и безопасности.

---

## 1. Полностью отсутствующие темы

### 1.1. Лицензирование и юридическая чистота
* **Статус:** В репозитории отсутствует файл `LICENSE`. По умолчанию это означает, что код закрыт копирайтом (All Rights Reserved).
* **Влияние:** Исключает adoption проекта в open-source сообществе и использование в корпоративном секторе.
* **Рекомендация:** Внедрить модель Open-Core.
    * Для бесплатного ядра репозитория: `Apache License 2.0` (предпочтительнее, защищает от патентных претензий) или `MIT`.
    * Для проприетарных плагинов (например, 3D-визуализатор, коннекторы): Закрытая коммерческая лицензия (EULA).

### 1.2. Локализация и मुльтиязычность (FTS5 + Кириллица)
* **Статус:** Не настроен токенизатор SQLite FTS5 для русского языка.
* **Влияние:** Дефолтный токенизатор `ascii` некорректно обрабатывает регистр кириллицы (поиск по "Агент" не найдет "агент") и не учитывает словоформы.
* **Рекомендация:** При инициализации базы данных жестко прописать `CREATE VIRTUAL TABLE ... USING fts5(..., tokenize='unicode61 remove_diacritics 1');`. Это решит 90% проблем с регистром и диакритикой без установки внешних библиотек.

### 1.3. Производительность на реальных объемах данных
* **Статус:** Отсутствуют бенчмарки для больших объемов (5000+ страниц).
* **Влияние:** Команда `/reindex`, использующая построчный `INSERT`, приведет к I/O bottleneck и задержкам в 3-5 секунд при росте базы.
* **Рекомендация:**
    * Использовать `cursor.executemany()` для пакетной вставки данных.
    * Обернуть переиндексацию в единую транзакцию `BEGIN TRANSACTION; ... COMMIT;`.
    * Внедрить инкрементальное обновление кэша (только измененные файлы по `mtime`).

### 1.4. Disaster Recovery и бэкап-стратегия
* **Статус:** Данные защищены только механизмом `git push`.
* **Влияние:** Если агент или пользователь случайно удалит локальные Markdown-файлы, а `push` еще не произошел, данные будут потеряны безвозвратно. Повреждение `wiki.db` не критично (это кэш), но потеря source of truth (файлов) фатальна.
* **Рекомендация:** Написать простой скрипт локального бэкапа, который по крону (раз в 4-6 часов) создает `tar.gz` архивы папки `.git` и `projects/` и складывает их в защищенную директорию с ротацией (хранить последние 10 копий).

### 1.5. Accessibility и UX для нетехнических пользователей
* **Статус:** Интерфейс ограничен CLI, что комфортно для разработчиков и агентов.
* **Влияние:** Высокий порог входа для менеджеров, аналитиков или исследователей, желающих использовать базу.
* **Рекомендация:** В долгосрочной перспективе (Phase 2+) — разработка легковесного Read-Only дашборда на FastAPI + HTML, который просто читает данные из `wiki.db` и предоставляет удобный поиск в браузере.

## 2. Поверхностно затронутые темы

### 2.1. Mobile capture-flow (Телеграм-бот + Pipedream)
* **Проблема безопасности:** Если Webhook на сервере принимает данные от Pipedream без строгой аутентификации, он уязвим для DDoS или мусорных инъекций извне. Утечка токена Pipedream компрометирует конвейер ввода.
* **Решение:**
    * Жесткая проверка `X-API-Key` в вебхуке на принимающей стороне.
    * *Альтернатива:* Убрать Pipedream и запустить локальный Telegram-бот скрипт (через `aiogram`) на сервере, использующий Long Polling. Это полностью закрывает данные внутри вашего контура.

### 2.2. Obsidian-интеграция и конкурентность
* **Проблема:** Возможные гонки данных (Race Conditions). Если Obsidian держит файл открытым на Windows, а автономный ИИ-агент пытается перезаписать его через `/save`, произойдет `PermissionError`.
* **Решение:**
    * Добавить папку `.obsidian/` в глобальный `.gitignore`.
    * В скрипт `reefiki.py` внедрить механизм файловых блокировок (например, библиотеку `filelock`) для безопасной параллельной записи.

### 2.3. Эволюция схемы данных (Migrations)
* **Проблема:** При обновлении `_schema.md` (добавлении новых полей во frontmatter) все старые страницы перестают проходить валидацию. Ручное обновление сотен страниц невозможно.
* **Решение:** Разработка команды `/migrate`. Использовать библиотеку `ruamel.yaml` вместо `PyYAML` — она умеет парсить и сохранять YAML без потери оригинального форматирования и комментариев пользователя, аккуратно дописывая новые дефолтные поля.

### 2.4. Конкурентный анализ в нише PKM
* **Notion AI:** Монолит, vendor lock-in. Агентам тяжело читать данные вне проприетарного API.
* **Logseq / Roam Research:** Отличные локальные системы, но они **Human-First**. Их разметка избыточна и сложна для программного парсинга.
* **Позиционирование REEFIKI:** Это **Agent-First** Knowledge Control Plane. Главное УТП — строгая типизация (через frontmatter `useful_when`) и абсолютная предсказуемость структуры, идеально подходящая для RAG и контекста LLM.

### 2.5. Метрики здоровья базы знаний (KPIs)
* **Проблема:** Отсутствует понимание того, насколько "замусорена" база знаний.
* **Решение:** Расширить команду `/status`, добавив блок `--health`, который вычисляет:
    * **Orphaned Nodes:** Количество `concept` или `entity` файлов, на которые нет ни одной ссылки из других документов.
    * **Quarantine Bloat:** Процент черновиков, превысивших лимит 30 дней.
    * **Distillation Ratio:** Отношение объема сырого текста (inbox/memoir) к структурированному (wiki/graphify).
