# Архитектурное предложение: Паттерны OpenViking для REEFIKI

Проект OpenViking (ByteDance/Volcengine) содержит эффективные архитектурные решения для управления контекстом AI-агентов. Поскольку REEFIKI является системой управления знаниями (PKM) и wiki-as-code, внедрение нижеописанных концепций значительно улучшит навигацию агентов по мультипроектной базе.

## 1. Парадигма виртуальной файловой системы (URI-адресация)

*   **Суть (OpenViking):** Отказ от классических ID векторной базы в пользу строгой URI-маршрутизации (например, `viking://resources`, `viking://agent`).
*   **Внедрение в REEFIKI:** Введение схемы `reefiki://` для всего содержимого.
    *   *Примеры:* `reefiki://projects/dashboardtt/architecture`, `reefiki://agents/jarvis/soul`.
*   **Зачем:** AI-редакторы (Cursor, Windsurf) и автономные агенты лучше итерируют по древовидной структуре. Интерфейс позволяет агенту запросить листинг директории (`ls`), чтобы понять структуру доступных ресурсов перед их чтением.

## 2. Уровневая загрузка контекста (Tiered Context: L0/L1/L2)

*   **Суть (OpenViking):** Тройная структура хранения сущностей:
    *   **L0:** Abstract (Краткая суть / TL;DR)
    *   **L1:** Overview (Оглавление / Ключевые поинты)
    *   **L2:** Full Content (Полный markdown / исходный код)
*   **Внедрение в REEFIKI:** При парсинге `.md` файлов автоматически генерировать слои L0 и L1. По умолчанию отдавать агенту только L0.
*   **Зачем:** Решает проблему "взрыва контекста". При генерации задач (например, для ночной смены) агенту не нужен весь документ на 5000 строк. Он получает мета-сводки (L0) нескольких проектов и проваливается на уровень L2 только при необходимости. Критически важно для Infinite Memory.

## 3. Директорийно-рекурсивный поиск (Directory Recursive Retrieval)

*   **Суть (OpenViking):** Семантический RAG-поиск ограничен деревом конкретных директорий, а не выполняется по всей плоской базе.
*   **Внедрение в REEFIKI:** Связать векторную базу с URI-путями. Поиск должен принимать параметр `scope` (область видимости).
*   **Зачем:** В мультипроектной вики контексты изолированы. Агент делает RAG-запрос исключительно внутри ветки `/metrica-gamma/`. Технические метрики одного проекта не смешиваются с лором другого (например, `/mandela-effect/`). Точность выборки радикально возрастает.

## 4. Унификация памяти, ресурсов и навыков (Unified Context)

*   **Суть (OpenViking):** Хранение всех типов данных под одним зонтиком файловой парадигмы.
*   **Внедрение в REEFIKI:** Интеграция в единое дерево не только заметок, но и:
    *   *Навыков:* Системные промпты, протоколы самовосстановления агента.
    *   *Ресурсов:* Справочные материалы (документация по Tonstakers, кошелькам и т.д.).
    *   *Памяти:* Логи прошлых сессий и эпизодическая память.
*   **Зачем:** Ядро идентичности (`SOUL.md`), техническая документация и рабочая память находятся в единой среде. Агенту не нужно переключаться между разными БД.

## 5. Нативная раздача базы через Model Context Protocol (MCP)

*   **Суть (OpenViking):** Встроенный MCP-сервер, оборачивающий логику RAG и VFS (виртуальной файловой системы).
*   **Внедрение в REEFIKI:** Архитектура REEFIKI должна изначально предоставлять интерфейс MCP-сервера.
*   **Зачем:** Мгновенная совместимость с AI-инструментарием. Claude Code, Windsurf или кастомные клиенты "из коробки" получают инструменты (`read_resource`, `search_directory`) для взаимодействия с персональной базой знаний.

## 6. Прозрачность траектории поиска (Context Observability)

*   **Суть (OpenViking):** Визуализация и логирование пути поиска агента (какие уровни и файлы были открыты).
*   **Внедрение в REEFIKI:** Логирование доступа агентов по путям `reefiki://` с указанием загруженных слоев (L0-L2).
*   **Зачем:** Упрощает отладку автономных систем (например, CoralOS). Если агент галлюцинирует, лог покажет, что он остановился на уровне L1 и не загрузил полное техническое описание (L2) перед началом кодинга.
