## 1. Как считать токены в REEFIKI

### 1.1 Типы расхода — они принципиально разные

```
ПОСТОЯННЫЙ расход (каждая сессия, неизбежен):
  Bootstrap — чтение контрактов и индексов при старте

ОПЕРАЦИОННЫЙ расход (за конкретную команду):
  /save, /process, /query, /harvest, /skill-port и т.д.

НАКОПИТЕЛЬНЫЙ расход (растёт с размером wiki):
  Зависит от N страниц, N источников, N паттернов
```

Ключевое отличие Claude Code от других IDE: **Claude Code передаёт
файлы которые явно открыты или запрошены, плюс system prompt**.
Он не грузит весь репозиторий в контекст автоматически.
Это значит bootstrap расход определяется тем что AGENTS.md
инструктирует агента прочитать при старте — не размером репозитория.

### 1.2 Единица измерения

Все цифры — **токены на операцию** (input + output вместе).
Для Claude Code: input tokens стоят дороже output при billing,
но для оценки нагрузки на качество важна сумма.

Типичный размер файлов REEFIKI:
```
AGENTS.md (корневой, 23KB):        ~5 500 токенов
COMMANDS.md (33KB):                ~8 000 токенов
CLAUDE.md (stub):                  ~50 токенов
projects/<name>/AGENTS.md:         ~2 000 токенов
projects/<name>/_domain.md:        ~300 токенов
wiki/index.md (100 страниц):       ~3 000 токенов
wiki/index.md (500 страниц):       ~12 000 токенов
wiki/index.md (1000 страниц):      ~22 000 токенов
Одна wiki-страница (средняя):      ~400–800 токенов
Один source файл (skills-core):    ~300–500 токенов
```

## 4. После всех описанных оптимизаций

Оптимизации из аудитов разбиты на три группы по механизму.

### 4.1 Группа O1 — Кэши вместо чтения файлов

**Drift cache** (из раздела оптимизации токенов предыдущих аудитов):
```
Было:  читать все source страницы → ~15 000 токенов
Стало: читать .drift-cache.json  → ~200 токенов
Экономия: 14 800 токенов/сессию
```

**Compiled user context** (_user/.compiled.json):
```
Было:  читать 3 файла _user/     → ~950 токенов
Стало: читать .compiled.json     → ~120 токенов
Экономия: 830 токенов/сессию
```

**index.min.json** вместо wiki/index.md:
```
Было (500 стр):  wiki/index.md   → 14 000 токенов
Стало:           index.min.json  → ~2 500 токенов
Экономия: 11 500 токенов/сессию  ← самая крупная
```

### 4.2 Группа O2 — Lazy loading

Агент НЕ читает при старте:
- COMMANDS.md (8 000 токенов) — только при явном запросе
- Полные wiki страницы — только при /query результате
- skills-core pages — только при /skill-port
- _user/log.md — только при записи

```
Экономия lazy loading:  ~8 500 токенов/сессию (COMMANDS.md + misc)
```

### 4.3 Группа O3 — Компактные форматы

AGENTS.md рефакторинг — убрать избыточные пояснения,
оставить только инструкции (аудит отмечает 23KB как "перегружено"):

```

Цель:  AGENTS.md 3 000 токенов (только директивы, без объяснений)
Экономия: 2 500 токенов/сессию
```

_user/profile.md компактный формат:
```

```

## 5. Дополнительная оптимизация без потери качества

Это то, что не описано в аудитах, но реально применимо.

### 5.1 Иерархический AGENTS.md — самый крупный незатронутый резерв

**Проблема:** AGENTS.md читается целиком при каждом старте.
Но большинство правил нужны только для конкретных операций.

**Решение — split по частоте использования:**

```
AGENTS.md (корневой) → только то что нужно КАЖДУЮ сессию:
  - bootstrap checklist
  - запреты (NEVER)
  - _user/ правила
  - ссылки на расширенные файлы

AGENTS-ops.md → операционные правила (читается при /save, /process):
  - save workflow
  - process workflow
  - harvest rules

AGENTS-governance.md → governance (читается при /lint, /resolve):
  - frontmatter schema
  - quarantine rules
  - conflict resolution

AGENTS-publish.md → публикация (читается при publish-task):
  - dual-remote rules
  - public/private classification
```

```
Было:  AGENTS.md 5 500 токенов каждую сессию
Стало: AGENTS-core.md ~1 500 токенов каждую сессию
       + профильные файлы по ~1 000 только когда нужны
Экономия на bootstrap: ~4 000 токенов
Потеря качества: нет — агент читает нужное правило перед операцией
```

### 5.2 Контекстное сжатие wiki/index

**Проблема:** при /query агент читает index, потом конкретные страницы.
При 1000 страниц index содержит много нерелевантного для конкретного запроса.

**Решение — FTS-pre-filter перед загрузкой в контекст:**

```python
def get_context_for_query(query: str, project: Path,
                           max_pages: int = 5) -> str:
    """
    Возвращает только релевантные страницы для конкретного запроса.
    Агент получает pre-filtered контекст вместо всего index.
    """
    results = search_existing(project, query, limit=max_pages)
    pages_content = []
    for result in results:
        page_path = project / "wiki" / result["id"]
        fm, body = parse_frontmatter(page_path.read_text())
        # Только frontmatter + первые 200 слов body
        pages_content.append(
            f"# {fm['title']}\n"
            f"useful_when: {fm.get('useful_when', '')}\n"
            f"tags: {fm.get('tags', [])}\n\n"
            f"{' '.join(body.split()[:200])}\n"
        )
    return "\n---\n".join(pages_content)
```

```
Было:  /query = bootstrap + весь index + полные страницы
Стало: /query = bootstrap + 5 pre-filtered страниц (усечённые)
Экономия: ~6 000 токенов на /query при 500+ страницах
Потеря качества: минимальная — FTS уже отбирает релевантное
```

### 5.3 Session memory — не перечитывать то что уже прочитано

**Проблема:** в длинной сессии с несколькими командами AGENTS.md
и project AGENTS.md читаются снова при каждой команде.

**Решение:** явная метка в начале сессии:

```markdown
## В AGENTS.md добавить:

### Session state
При старте сессии прочитай все обязательные файлы ОДИН РАЗ.
Сохрани в memory:
  session_bootstrapped: true
  bootstrapped_files: [список]

При последующих командах в той же сессии:
  НЕ перечитывай файлы из bootstrapped_files
  ЧИТАЙ только файлы специфичные для команды
```

```
Экономия: bootstrap читается 1× вместо N× за сессию
При сессии с 10 командами: экономия 9 × 9 320 = 83 880 токенов
```

Это самая крупная оптимизация из дополнительных.
Работает потому что Claude Code поддерживает контекст
в рамках одной сессии.

### 5.4 Diff-only updates для patterns.md

**Проблема:** при обновлении _user/patterns.md агент
перезаписывает файл и читает его снова.

**Решение:** только append, никогда полная перезапись:

```python
def append_pattern(user_dir: Path, observation: str,
                   confidence: str = "low") -> None:
    """Добавляет наблюдение в конец файла без перечитывания."""
    from datetime import date
    entry = (
        f"\n[{date.today()}] {observation} "
        f"confidence: {confidence}\n"
    )
    with (user_dir / "patterns.md").open("a",
          encoding="utf-8", newline="\n") as f:
        f.write(entry)
    # Инвалидировать compiled кэш
    compiled = user_dir / ".compiled.json"
    if compiled.exists():
        compiled.unlink()
```

Агент не перечитывает весь файл — только добавляет строку.

### 5.6 Граница: что нельзя оптимизировать без потери качества

Это важно — не всё поддаётся сжатию:

```
НЕЛЬЗЯ сжимать без деградации:
──────────────────────────────────────────────────────────────
/process решение            — требует полного контекста страницы
/skill-port адаптация       — требует обоих файлов целиком
/resolve конфликт           — требует обоих conflicting источников
Governance правила (AGENTS) — нельзя сокращать запреты
useful_when validation      — требует полного поля для оценки

МОЖНО сжимать без деградации:
──────────────────────────────────────────────────────────────
Index для навигации         → index.min.json
Drift detection             → .drift-cache.json
User context                → .compiled.json
AGENTS.md объяснения        → убрать "почему", оставить "что"
COMMANDS.md при старте      → lazy load
Повторный bootstrap         → session memory
```

## 7. Практические рекомендации

### 7.1 Что сделать прямо сейчас (Phase 0i, 100 страниц)

Сейчас wiki маленькая, проблема не критична. Но заложить правильный
фундамент дешевле чем переделывать при 500 страницах:

```
1. Добавить в AGENTS.md: "читай файлы 1× за сессию, не повторяй"
   → Экономия session memory, 0 усилий на реализацию

2. Реализовать index.min.json в /process pipeline
   → Главная проблема роста решена превентивно

3. Создать .drift-cache.json скрипт + git hook
   → При 10 sources уже заметно; при 30 критично

Время: 1–2 дня. Блокирует деградацию при росте wiki.
```

### 7.2 Что сделать при 200–300 страницах


### 7.4 Формула для самопроверки

Если хочешь оценить реальный расход до/после изменения:

```python
# Быстрая оценка токенов любого файла:
def estimate_tokens(file_path: str) -> int:
    text = open(file_path, encoding="utf-8").read()
    # Приближение: 1 токен ≈ 4 символа EN, ≈ 2.5 символа RU/mixed
    return len(text) // 3  # консервативная оценка для mixed RU/EN
```
